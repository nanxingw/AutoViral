Module 1: WsBridge — Agent Session Manager

Core engine replacing the current executor. Based on pneuma-skills pattern.

WsBridge Lifecycle
Session Lifecycle
① User creates new Work
  → wsBridge.createSession(workId)
  → spawn claude --sdk-url ws://localhost:3271/ws/cli/{workId}
  → CLI connects back as WebSocket client
② CLI sends system.init
  → save cliSessionId for resume
  → session ready for messages
③ Backend sends initial prompt
  → NDJSON: {"type":"user","message":{...}}
  → includes memory context from EverMemOS
④ Agent streams response
  → stream_event tokens → frontend WebSocket
  → result = turn complete
⑤ User sends feedback/refinement
  → same WebSocket, next turn
  → Agent has full conversation context
⑥ User closes or server restarts
  → persist cliSessionId to work store
  → next open: claude --resume {cliSessionId}
Message Flow
Browser → Backend → CLI
Browser WS: {"action":"send","text":"改一下标题"}
↓ Backend translates
CLI WS: {"type":"user","message":{"role":"user","content":"改一下标题"}}
CLI → Backend → Browser
CLI WS: {"type":"assistant","message":{"content":[{"type":"text","text":"..."}]}}
↓ Backend forwards
Browser WS: {"event":"agent_message","data":{"text":"..."}}
Key Difference from Current Executor
Current: spawn → single prompt → parse stdout → done
New: spawn → persistent WS → multi-turn → resume
The CLI process stays alive for the entire work session. User can send unlimited messages. Full Claude Code capabilities (tool use, file ops, MCP) available throughout.

WsBridge API (new module: src/ws-bridge.ts)

Method Purpose
createSession(workId, initialPrompt) Spawn CLI, return session object. Injects EverMemOS memory into prompt.
resumeSession(workId) Respawn CLI with --resume using stored cliSessionId.
sendMessage(workId, text) Send user message to active CLI session via WebSocket.
killSession(workId) Gracefully terminate CLI process, persist sessionId.
getSession(workId) Return session state: idle/busy, message history, cliSessionId.
onEvent(workId, handler) Subscribe to CLI events (stream tokens, tool use, result).

Coexistence with Existing Executor

The existing executor.ts continues to handle evolution cycles (evo-context, evo-skill, evo-task) via stdout parsing. WsBridge is only for content creation sessions where multi-turn dialogue is needed. They share the same Hono server and WebSocket upgrade path.

Route: /ws/cli/:workId → WsBridge (CLI connects here)
Route: /ws/browser/:workId → WsBridge (frontend connects here)
Route: /ws → existing executor broadcast (unchanged)
