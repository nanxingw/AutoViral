Revised Architecture: Claude CLI WebSocket (pneuma-skills pattern)

Based on your discovery — persistent multi-turn conversations via Claude CLI's --sdk-url

Core Innovation

Instead of spawning a new CLI process per step (current approach), we maintain a persistent Claude CLI session per work. The Agent remembers everything about this content piece across all 6 steps + user dialogue.

Content Creation Flow (per Work)
User: New Work → Spawn CLI Session → Step 1: Topic ⇄ User Chat → Step 2: Remix ⇄ ... → Publish → Collect Metrics → Update Memory
Key Upgrade: Executor → WsBridge
Before: executor.run() → spawn CLI → parse stdout → one-shot result
After: wsBridge.createSession() → spawn CLI with --sdk-url → persistent WebSocket → multi-turn dialogue
Resume: CLI session persisted → --resume on restart → full context restored
Memory: Each work's conversation history = creation context for next work

Sub-Systems Breakdown

1. Content Engine
  • WsBridge (persistent CLI sessions)
  • Work Store (YAML, per-work state)
  • Pipeline orchestration (6 steps + custom)
  • Chat interface (user ⇄ Agent dialogue)
  • Multi-format (video script, image-text, livestream)
2. Publish Engine
  • Playwright browser automation
  • Platform adapters (Xiaohongshu, Douyin)
  • Login state management (cookies)
  • Content formatting per platform
  • Scheduling (publish at optimal time)
3. Data Collector
  • Playwright periodic scraping
  • Metrics: views, likes, comments, shares
  • Trend data (real, not mock)
  • Competitor content tracking
  • Cron-scheduled collection jobs
4. Memory System
  • Creation history (what worked, what didn't)
  • Style profile (evolving, data-driven)
  • Platform rules (learned from successes/failures)
  • Competitor insights (accumulated)
  • Injected into CLI session as system context
