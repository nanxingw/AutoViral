Architecture Approaches

How should we build the full content creation + publishing platform?

A

Extend Existing Executor (Recommended)

Core idea: Reuse the current Claude CLI executor pattern. Each pipeline step = a job that spawns claude CLI with a specialized prompt.

User clicks "Start Creation"
  Step 1: executor.run() → Claude CLI → topic generation
  Step 2: executor.run() → Claude CLI → trend remix
  ... (each step streams progress via WebSocket)
  Step 6: playwright.run() → browser automation → publish
Post-publish: executor.run() → scrape metrics → update memory

Pros

  • Reuses proven executor + WebSocket infrastructure
  • Minimal new code — extend, don't rebuild
  • Claude CLI handles auth, streaming, tool use natively
  • Each step is an independent job (resilient to failures)

Cons

  • Multi-turn dialogue needs session state management
  • CLI spawning has overhead (~2s per launch)
  • No built-in conversation memory between steps
B

Claude API Direct Integration

Core idea: Replace CLI executor with direct Claude API calls via @anthropic-ai/sdk. Enables native multi-turn conversations within each step.

User clicks "Start Creation"
  Session: anthropic.messages.create() with full conversation history
  User feedback → append to messages[] → regenerate
  ... (true multi-turn dialogue per step)
  Publish: playwright.run()

Pros

  • Native multi-turn conversation (no session hacks)
  • Lower latency (no CLI spawn overhead)
  • Full API control (tool use, vision, etc.)

Cons

  • Requires API key management (not using Claude Code auth)
  • Must rebuild streaming, tool use, error handling from scratch
  • Loses Claude Code's built-in MCP/skill capabilities
C

Hybrid: CLI + Conversation Layer

Core idea: Use CLI executor for one-shot generation, add a lightweight API conversation layer for interactive revision. Best of both worlds.

Step auto-generation:
  executor.run() → Claude CLI → initial result
User wants to refine:
  conversation.send() → Claude API → revised result
User satisfied → next step:
  executor.run() → Claude CLI (with previous context)

Pros

  • Best of both: CLI tools for generation, API for dialogue
  • Multi-turn revision is natural
  • Can progressively migrate from CLI to API

Cons

  • Two integration points to maintain (CLI + API)
  • Context sync between CLI and API conversations
  • More complex architecture
