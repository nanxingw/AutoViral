**为何 ready-for-human**：修法是架构选择——(a) resume 时 `--append-system-prompt` 注入增量教学（CLI 是否支持需验证）；(b) 提示词版本号写进 work.yaml，过期则放弃 resume 开新 session（丢上下文换新认知）；(c) 把易变教学从提示词挪进 `autoviral docs`（resume 也会重查 docs），提示词只留稳定骨架。三者取舍涉及上下文保留 vs 认知新鲜度，需用户拍板。

**Code-area hints**：`src/ws-bridge.ts`（createSession/sendMessage resume 分支）。

**2026-06-10 决策注记（选型可追溯 / 可否决）**：
- **背景**：B7 原标 `ready-for-human`（修法是架构选择，需用户拍板）。用户 2026-06-10 下达「修复所有 bug」授权，并在侦察实证（claude CLI 2.1.170 确有 `--append-system-prompt` 与 `--fork-session`、与 `-p`/`--resume` 无互斥；claude-api skill 实证 mid-conversation system 通道是 Anthropic 官方推荐的「不破缓存、就地补教学」正解，且要求增量写成 *context, not commands*）后由编排层选型，理由记此供用户追溯/否决。
- **选定方案：(c) 教学下沉 docs 为主 + (a)-lite resume 增量注入为辅；(b) 不采用。** 三轴论证：
  - **上下文保留**：(c) 满分（resume 照常续接 CLI 上下文）；(a) 满分（`--append-system-prompt` 就地补，prefix cache 不破）；(b) 最差（过期放弃 resume 开新 session = 丢光 CLI transcript，旧 work 用户每次升级后第一句都像跟陌生 agent 重来）。
  - **认知新鲜度**：(a) 最高（每次 resume 注入最新增量）；(c) 中高（manual 最新，靠 agent 主动 `autoviral docs` 查 + 提示词留稳定骨架句把命中率拉满）；(b) 最高但代价大。
  - **实现风险**：(c) 最低（纯文档迁移 + 提示词删段留指针，不碰 spawn/session）；(a) 中（要去重「已注入到哪个版本」，增量须 context-not-commands）；(b) 中（work.yaml 加版本 + 多会话存哪、每次丢上下文是产品体验代价）。
  - **为何 (b) 弃用**：丢上下文代价不可接受——多轮 work 的连贯性是产品核心，仅在「提示词骨架级不兼容重写」的极端情形才可能作兜底，本批不触发。
- **实现落点**：
  - **(c) 主线（B7 与 B4 一体）**：`src/ws-bridge.ts` `buildSystemPrompt` 素材生成段瘦身为「端点名册」（capability + method + path，让 agent 知道有什么）+ 段尾稳定骨架句「所有素材生成端点参数以 `autoviral docs _shared/03-cli-reference` 为准——现查 docs，不要凭记忆里的旧参数」。**参数表全部移出提示词、只活在 manual**（B4 已迁齐 tts/captions/mix/transitions/frames）。B2 加的兜底禁令（不存在的 `.py` → `autoviral ask`）作为稳定纪律保留在提示词。`ws-bridge.test` 与 `docs-drift.test` 的提示词锁有意识改写：语义从「prompt 含参数表」改为「prompt 含名册 + docs 指针，manual 含参数表」，绝不为过测试留双份参数表。
  - **(a)-lite 辅线**：`src/ws-bridge.ts` 新增 `PROMPT_VERSION` 常量 + `PROMPT_CHANGELOG` map（version → 简短 *陈述句* context，措辞实证过 Opus 对 system 通道命令式 override 有抵抗，故全用陈述句不用命令式）。sidecar `SessionRecord` 加 `lastInjectedPromptVersion`（per-session，与 cliSessionId 同层，因 ADR-008 每 session 各有 CLI lineage 各自落后）。`spawnCli` 加 `--append-system-prompt` 形参；resume 分支经 `resumePromptAppend(workId, sid)` 计算「自 stored 版本以来的 changelog 拼接」并传入，成功后把 stored 推进到 `PROMPT_VERSION`；fresh 分支在 `system.init` 写回时直接记当前版本。**仅对 work-bound 会话生效**（coach/trends 跑别的提示词，helper 内 `isWorkBound` 守卫排除）。单测覆盖：落后→args 含 `--append-system-prompt` 且内容来自 changelog；持平→不含；版本推进正确；fresh 记版本；legacy 无版本→注全量。
- **未实测/风险**：`--append-system-prompt` 在 `--resume` 下把追加段插进既有 transcript 的**实际插入位置与生效性**未起真会话探针（侦察确认 flag 存在且无互斥，实现按文档行为做）；E2E 由后续 Workflow 多纬度验。
- **用户可否决**：若用户认为旧 work 该用 (b)「过期即开新 session 换全新认知」，可重开此片切换方案——本注记保留三轴论证以便重评。

**Wave B review fixup（2026-06-11）**：
- **版本戳竞态/泄漏修复**（review MEDIUM）：`system.init` / `result` 两处写回原**无条件**把 `lastInjectedPromptVersion` 盖成 `PROMPT_VERSION`，即使本次 resume 实际跳过了注入——会把「未注入」误记为「已教到 current」，永久压制该 session 的教学。改法：`WsSession` 加 `taughtPromptVersion`（本次 spawn 真正教到的版本），`resumePromptAppend` 按实际投递路径设它（注入了 → current；已持平 → stored；读异常 → 不设，writeback 跳过 stamp），fresh 分支设 current；两处 writeback 改为只 stamp `session.taughtPromptVersion`（undefined 时不写）。附带的 **downgrade 防护**：被更新 build 教到更高版本的 record 不再被旧 server 下压到 `PROMPT_VERSION`。新增单测：future-version record resume + `system.init` 后版本不被下压。
- **「真·无 record legacy」LOW 复核**：B4-team 担心的无-record 路径在 `createSession` 实际不可达——`ensureSidecarRecord`（:594）在 resume 分支（:640）前就为默认会话从 work.yaml 播好了 record，故 `resumePromptAppend` 读到的 record 必存在、版本能持久化。`if (record)` 守卫仅作防御，已在 docstring 注明。
- **PROMPT_CHANGELOG 含枚举碎片**（review LOW）：接受为已知项——changelog 是给 resumed 会话补的简短 *陈述句* context，含 `durationSec 4–15` 等碎片是有意的「告诉 agent 有什么变了、详见 docs」，不受单一事实源锁约束。已在 `PROMPT_CHANGELOG` docstring/注释说明它独立于 `buildSystemPrompt`、不被 `not.toContain('durationSec')` 锁覆盖，避免后续 reviewer 误判违纪。
