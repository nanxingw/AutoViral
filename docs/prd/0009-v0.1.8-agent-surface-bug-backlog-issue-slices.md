## B1 · 聊天暂停键无效 —— abort 端点 session-blind（pre-ADR-008 化石）`ready-for-agent`

**What**：AI 流式输出时按输入框右下角红色停止键无任何效果（用户 2026-06-10 报告，多会话 work 实测）。

**根因（已钉死）**：
- 前端 `web/src/features/studio/panels/Chat/index.tsx:644-649` → `POST /api/works/:id/abort`，**不带 sessionId**；
- 端点 `src/server/routes/works.ts:451-457` → `wsBridge.killSession(id)`，**不传第二参**；
- `src/ws-bridge.ts:903-905` `killSession(workId, sessionId?)` → `resolveSessionId(undefined)`（:373-374）→ 永远落 `DEFAULT_CHAT_SESSION_ID = "s_1"`（:85）。

**Acceptance criteria**：
- [ ] 前端 abort 请求携带当前活跃 sessionId；端点接收并透传 `killSession(workId, sessionId)`。
- [ ] `{aborted:false}` 时 UI 给出可见反馈（toast/错误态），不再静默。
- [ ] 测试：s_2 流式中 abort → s_2 的 cliProcess 收到 SIGTERM 且 `session_killed` 只广播给 s_2 的 socket，s_1 不受影响；单会话 work 回归不破。
- [ ] E2E 验收经 Workflow 多纬度 subagent 执行（双会话纬度必测）。

**What**：chat-s_2 实证两个缺口逼 agent 直编 yaml：①`autoviral comp set --duration` 无此选项（comp.duration 不自动推导的老坑，MEMORY 在案）；②clip 跨轨移动 bridge PATCH trackId → 400 不可 patch，直编 yaml 又被 daemon 内存态回写还原（file vs server 状态分歧陷阱），agent 只能放弃并口头让用户去 UI 手挪——而 UI 侧 #88 已有 moveClipToTrack store action，bridge `routes.ts:1583` 有 `POST /clip/:id/move` 待确认是否支持换轨。agent-人对等性（产品命题核心）在这两处破口。

**Acceptance criteria**：
- [x] `autoviral comp set --duration <s>`（或自动推导）落地；
- [x] clip 换轨经 bridge/CLI 可达（复用 #88 的 kind-guard 语义），`autoviral clip move <id> --to-track <trackId>`；
- [x] 「直编 yaml 被内存态回写」的行为在 manual 写明（agent 必须走 bridge 写入的原因）。

**Code-area hints**：`cli/autoviral/src/commands/{comp,clip}.ts`、`src/server/bridge/routes.ts`。

**2026-06-10 实施注记**：
- **前提一半已过时（clip 跨轨移动早已全链路存在，不是缺口）**：本片侦察证实 ①What 里「bridge PATCH trackId → 400 不可 patch / UI-only」描述失实。clip 换轨 CLI→bridge→ops 全链路从 S8 起就在：CLI `autoviral clip move <id> --to-track <trackId>`（`cli/autoviral/src/commands/clip.ts:125-150`）→ bridge `POST /clip/:id/move`（`src/server/bridge/routes.ts:1583`，走 `ops.moveClipToTrack` 的 same-kind guard / trackOffset 保留 / 源轨 orphan-transition 清理，与 Studio #88 共用同一 SHARED op）→ 已有 S8 路由测试覆盖（`routes.test.ts:3103-3327`）。本片**不重造**，只复核 + 补 manual 文档。
- **PRD 命令签名笔误已纠**：原 AC 写 `clip move <id> --track <trackId>`，实现是 `--to-track`（`--track` 在 `clip add` 里是 *kind* 语义，复用会歧义）。文档对齐为 `--to-track`，**不加别名**（零代码风险，避免两个语义重叠 flag）。
- **真缺口 = `comp.duration` 无写入路径，已修**：CLI `comp` 子命令原只有 show/diff/put/validate/aspect，无 `set`；ops 集合无 `setDuration`；web store 只单调增长（`store.ts` 多处 `Math.max`），缩短整体时长无路径——这正是 chat-s_2 逼 agent 直编 yaml 的根因。本片新增：
  - SHARED 纯函数 `setCompositionDuration(comp, {durationSec} | {auto:true})` + `compositionContentEnd`（`src/shared/composition/ops/setDuration.ts`，index 导出）；显式秒数（有限非负，违规 `CompositionOpError code:4`）或 auto 从全部 track 的 clip end 取 `Math.max`（与 store clipEnd 同口径）。允许设小于内容 end 的值（缩短是合法意图）。
  - bridge `POST /comp/duration`（`routes.ts`，紧贴 `/comp/aspect`），body `{durationSec}` 或 `{auto:true}`，响应带 `{duration, contentEnd, truncatesContent}` 供 CLI 警示。
  - CLI `autoviral comp set --duration <seconds|auto>`（本地先校验，违规 exit 4 不打 bridge；`auto` 字面量→`{auto:true}`）；`truncatesContent=true` 时 stderr 打非阻断警告（时长短于内容 end，尾部不渲染）。`--help` 加一行。
- **「直编 yaml 被还原」真机制**：不是 server 内存缓存（server 每次读盘、无状态），而是 **Studio 编辑器打开该 work 时的 800ms 防抖整份 `PUT /comp` 自动保存**（`web/src/pages/Studio.tsx:192-211`）用编辑器内存态覆盖直编。仅在【编辑器对该 work 打开且内存 comp 变脏】时发生；纯 headless 直编不会被覆盖。已写进 manual `_shared/03-cli-reference.md` 的「⚠️ NEVER hand-edit composition.yaml」警告段（精确措辞 + 走 bridge 才安全且广播给 UI 的理由）。
