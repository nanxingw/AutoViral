## Problem Statement

AutoViral（`autoviral@0.1.0`）已经能发布、能渲染、能驱动 agent。但**当用户想"加一个新内容类型 / 新 provider / 换底座 agent"——也就是项目"扩展与升级"这件事——成本是反常地高的**，原因是它相对两个顶级范式存在结构性短板：

- **pneuma-skills** 把每个内容类型做成自包含插件（`modes/<name>/` = manifest + skill + viewer + seed + showcase），加内容类型 = drop 一个目录；
- **claude-code** 按职责垂直切片（`commands/` 每命令一目录、`bridge/` 每文件一职责），没有 god-module。

对照之下，AutoViral 的结构是**"两类内容类型各自硬连线、能力注册表分裂、文档治理半残"的早期工位形态**：

1. **内容类型写死成二元枚举，没有注册表抽象。** `WorkType = "short-video" | "image-text"` 向下散弹辐射到 server 的裸字符串分支、checkpoint 的 `DELIVERABLES` 硬编码列表、两套互不共享的 web 视图树（studio ~183 文件 vs editor ~53 文件）、以及一份只覆盖 video 的全局 skill 手册。加第三种内容类型今天是一次跨 5+ 文件的散弹手术 + 复制一整套视图树。这是用户"加新内容类型更容易"诉求的最大阻塞。

2. **provider 有三套并行机制。** image 走 `src/providers/registry.ts`（像样的 Map 注册表）、video 绕过它另起一套 `src/server/providers/registry.ts`（两个 registry 同名 `getProvider`，靠 import 别名区分）、TTS 又是第三条路（直连 endpoint 不经任何 registry）。这直接违反 CONTEXT 不变量 #2「provider 经 `src/providers/registry.ts` 注册」。加新 provider 的第一道认知负担是"先判断该往哪个 registry 加"。

3. **carousel 是一等内容类型却 skill + CLI 零覆盖（本版唯一 high 差距）。** 图文是 AutoViral 两大交付物之一，但 `skills/autoviral/` 手册对它 0 覆盖、`autoviral` CLI 对 `carousel/slide/layer` 0 命中。agent 编辑图文作品时唯一指引是 system prompt 里一行 prose（省略所有 union 形状/enum/box 约束），盲写必然产出 zod 校验不过的 yaml → 用户在 Studio 看不到图文。

4. **宣称 agent-agnostic 却无 `AGENT.md`。** CLAUDE.md / README 反复承诺 codex/kimi/gemini/aider 都能驱动，但仓库无 `AGENT.md`/`AGENTS.md`——非-Claude agent 进仓只有一份文件名+语境默认 Claude 的 `CLAUDE.md`。宣称与文档治理自相矛盾（pneuma 有 `CLAUDE.md` + `AGENT.md` 双轨 + 一张"Documentation Policy"映射表）。

5. **一批零风险工程债阻碍上手。** root 残留 Svelte 时代化石 `svelte.config.js`（项目早已 React 19，零 svelte 依赖）、孤儿 `web/package-lock.json`（无对应 manifest）、被 commit 进仓的 `.vite/deps/` 缓存、废弃 `test-studio.mjs`/`ecosystem.config.cjs`；发布构建把 92 个 `*.test.ts` 一起编进 `dist/`；web 最严 tsconfig（`noUnusedLocals` 等）从不在 CI 跑、且 `ci.yml` 有一条谎称 `tsc --noEmit` 已绿的假注释。

6. **源码结构有集中的可读性热点（导航层，非文件层）。** 单个 `src/server/api.ts` 是 **3270 行 / 80 路由 / 23 个不相干领域**的 god-module——占后端非测试代码总量（18523 行）的 **≈18%**，是断崖式离群点（第二大的 `ws-bridge.ts` 才 1165 行）。`src/` 根目录另平铺 **17 个混层 `.ts`**（基础设施 `config/logger/paths` + 领域 `work-store/memory/analytics` + agent 桥 + 测试编排全在同一层，无 `infra/`/`domain/` 归类）。注意这是**导航可读性**问题（"某路由在哪 / 新模块放哪要全文件搜"），不是文件内可读性问题（单函数大多清晰）；也**不是正确性/运行时缺陷**——故定级 medium、排在最后做。反证它可控：`src/server/bridge/routes.ts`（762 行）已经是一次干净的子 router 拆分，证明拆 `api.ts` 是照搬已验证模式的低风险机械活，不是推倒重来。

这些短板互相咬合：没有 registry 抽象，加内容类型就要散弹手术；散弹手术又因为缺 skill/CLI 覆盖而退化成"prompt 里塞一行裸 schema"；文档治理半残（无 AGENT.md、README drift、ADR 无索引）让新 agent/新 contributor 看不清真实地基；而 `api.ts` god-module 与混层 root 让"加 endpoint / 加模块该落在哪"缺乏锚点，与扩展诉求直接相悖。不一次性奠基，每加一种内容类型/provider 债务就超线性增长。

## Solution

**v0.1.1 的主线是"可扩展性奠基"**：不追求一步到位的完整 mode 插件系统（pneuma 那种 + 远程装 mode，是 0.2.0 级 XL 重构），而是**把"加内容类型 / 加 provider"从散弹手术降维成"往注册表加一条 + 实现一个契约"**。落地三个高 ROI 的深模块骨架，补齐"升级不丢数据"的迁移骨架，清掉一批零风险工程债，补齐文档双轨治理。

一次性交付（七个带 Gate 的 workstream，依赖排序）：

- **W1 仓库卫生清债** — 删化石/孤儿/缓存/废弃脚本，发布构建不再编测试文件。
- **W2 文档双轨治理** — 建 `AGENT.md`、README 去 drift、ADR 索引、Version Bump Checklist、docs-drift 守卫测试。
- **W3 CI 类型门** — web 最严 tsconfig 入 CI + 修假注释。
- **W4 ContentTypeRegistry（深模块 ①）** — `WorkType` 二元枚举 → 中央注册表 + seedFactory，消灭散弹裸字符串分支。
- **W5 单一 MediaProvider Registry（深模块 ②）** — image/video/TTS 三套并行 → 一个 capability-tagged 注册表，兑现不变量 #2。
- **W6 Skill co-location + carousel 覆盖** — 全局 video-centric 手册 → 按内容类型分子树，补 carousel schema 文档 + CLI 命令（关 high 差距）。
- **W7 模块边界拆分** — `api.ts` god-module 按域拆子 router；root 散落文件归类。
- **W7.5 升级骨架（深模块 ③，应用户测试诉求纳入）** — composition/carousel schema 加 `schemaVersion` 字段 + `src/shared/migrations/` 顺序迁移注册表骨架，收编现有临时迁移逻辑。

核心原则（CLAUDE.md「Boil the ocean」+「Evidence over agreement」）：能一次性永久奠基就奠基；每个 workstream 的"完成"必须有可观测 Gate（测试全绿 + 浏览器可见 E2E 截图）背书。骨架落地后，0.2.0 加第三种内容类型从"改 5 处 + 复制视图树"收敛为"加一个注册项 + 实现 viewer 契约"。

## 目标用户（for whom）

1. **维护者（你）想加第三种内容类型**（海报 / wallpaper / 长图文）——希望是"加目录/加注册项"，而非跨 5+ 文件的散弹手术。
2. **维护者想加新 provider**（新 image/video/TTS 厂商）——希望只有一个 registry、一套契约、一种 env-key 声明方式。
3. **AFK / 多 agent 并行实现者**——希望 root 干净、模块边界清晰、CI 有类型门兜底坏味道，并行写代码不互相踩。
4. **非-Claude CLI agent（codex/kimi/gemini/aider）**——希望进仓有 `AGENT.md` 项目级入口，兑现"agent-agnostic"的宣称。
5. **驱动图文作品的 agent**——希望有 carousel 的 schema 文档 + 原子 CLI 命令，不必凭一行 prose 盲写 yaml。
6. **未来升级数据格式的人**——希望有 `schemaVersion` + 顺序迁移纪律，旧 work 升级不静默坏掉。

### User Stories

1. 作为维护者，我想要内容类型由一个中央注册表声明，以便加第三种类型时只新增一个注册项而非改散落在 server/web 的 5+ 处裸字符串分支。
2. 作为维护者，我想要 checkpoint 的交付物清单（`DELIVERABLES`）从注册表派生，以便加内容类型时它自动包含新类型的产物文件，不必手改常量。
3. 作为维护者，我想要新建空白作品的"seed 模板"挂在注册项上（`seedFactory`），以便创建流程遍历注册表渲染类型选项，加类型不改 `NewWorkCard` / server 创建逻辑。
4. 作为维护者，我想要 image/video/TTS 三类 provider 收敛到单一 `MediaProvider` 注册表（capability 区分能力），以便加新 provider 只认一个入口、一套契约。
5. 作为维护者，我想要 provider 的 env-key 映射声明式集中（仿 pneuma manifest `envMapping`），以便"哪个能力用哪个 key"一眼可查、不散落。
6. 作为驱动图文的 agent，我想要 carousel 的完整 schema 文档（slides/layers discriminated-union/bg/globals），以便不必凭 system prompt 一行 prose 盲写、产出能过 zod 校验的 yaml。
7. 作为驱动图文的 agent，我想要 `autoviral carousel add-slide` / `set-layer` 等原子 CLI 命令（对齐 `clip add/set` 的 zod 校验模式），以便两类内容类型在 CLI 上对称、可走协议而非裸写文件。
8. 作为加载 skill 的 agent，我想要手册按内容类型分子树（`manual/_shared/` + `manual/{video,carousel}/`），以便编辑 carousel 时不被灌一整章 video composition schema。
9. 作为非-Claude CLI agent，我想要仓库有 `AGENT.md` 项目级入口，以便进仓就知道如何操作工位，兑现 README 的 agent-agnostic 承诺。
10. 作为新 contributor，我想要 README 的目录结构/架构图/provider 表与 0.1.0 现状一致，以便不照着已删除的 `modules/` 脚本路径或违反不变量的多 provider 列表去扩展。
11. 作为查阅架构决策的 agent，我想要 `docs/adr/` 有一张索引表（编号/标题/状态/日期 + 状态机），以便"质疑某不变量前先翻既有决策"一眼可查、新增 ADR 有强制登记点。
12. 作为发版的人，我想要 CONTRIBUTING 有枚举式 Version Bump Checklist，以便 bump 时不漏改 `CLAUDE.md` 里硬编码的版本行、不与 `package.json` 漂移。
13. 作为前端扩展者，我想要 web 最严 tsconfig（`noUnusedLocals` 等）在 CI 真正执行，以便未用变量/漏 case 这类坏味道在合 main 前被拦，而不是合入后才发现。
14. 作为新 agent 上手者，我想要 root 只剩活配置（删 svelte 化石/孤儿 lockfile/废弃脚本/commit 进仓的缓存），以便扫 root 判断技术栈/启动路径时不被误导。

17. 作为维护者，我想要 root 散落的 17 个混层 `.ts` 归类到 `src/{infra,domain}/`，以便新增模块有明确归属目录。
18. 作为升级数据格式的人，我想要 composition/carousel schema 带 `schemaVersion` 字段 + 顺序迁移注册表，以便旧 work 加载时按版本链式迁移、不静默坏掉。
19. 作为质量把关者，我想要 prompt + SKILL.md 里引用的所有 `manual/<file>` / `docs <topic>` 有一个 fs 契约守卫测试，以便手册重命名/拆分时悬空引用立刻变红，而非 agent 跑 `autoviral docs` 拿到静默 404。
20. 作为维护者，我想要 recipes 按内容类型分区（`recipes/{video,carousel}/`）+ 一两条 carousel recipe 范例，以便加新类型 recipe 有模板可循、不往平铺目录堆。
21. 作为 reviewer，我想要每个 workstream 的"完成"以浏览器可见 E2E 截图为准（不变量 #6），以便确认 registry 改写/手册重构/路由拆分没在"最后一公里"断掉用户可见行为。

## Implementation Decisions

> 遵循 `docs/prd/README.md`「PRD 不写怎么实现」与 to-prd「不含具体行号/代码片段（易过期）」：本节只锁定**模块边界、接口形状、架构决策**。精确的文件/行号锚点见 ## Further Notes 指向的 gap 分析。需锁定的架构选择独立成 ADR（见末尾建议）。

### 深模块 ① — ContentTypeRegistry（W4，主线核心）

- **深模块判据**：用一个少变接口（`getContentType(id)` / `listContentTypes()`）封装"每种内容类型的差异行为"，逻辑全在内部，加类型不改接口、纯函数可隔离测试。
- **接口形状**（决策，非最终代码）：注册项声明 `id`（`WorkType`）、`displayName`（i18n key）、`deliverableFile`、`routePath(workId)`、`seedFactory(workId)`、`checkpointTargets`、`schemaModule`。
- **改写方向**：`WorkType` 枚举从注册表派生；server 的 `workType !== "short-video"` / `!== "image-text"` 裸比较改查表；checkpoint 的 `DELIVERABLES` 改为 `listContentTypes().flatMap(t => t.checkpointTargets)`；video 初始 composition 拼装 + carousel `makeEmptyCarousel` 收编进各自注册项的 `seedFactory`；`NewWorkCard` 的写死两个类型选项改为遍历注册表渲染；`WorksGrid` 的 `?'/studio/':'/editor/'` 三元改 `getContentType(w.type).routePath`。
- **范围克制**：v0.1.1 用中央 Record 注册两条（short-video / image-text），**不追求 runtime 自动发现**——证伪已澄清 pneuma 自己也是 hardcoded `builtinModes` 中央 map，仅 external mode 动态。不强行统一 video（时间线/帧/transitions）与 carousel（静态 slide/layer）的领域 schema 本体——它们是真异构领域（见 Out of Scope G09）。

### 深模块 ② — 单一 MediaProvider Registry（W5）

- **接口形状**：单一注册表，provider 按 capability（`image | video | tts`）标记，`getProvider(capability, name)` / `getDefaultProvider(capability)` / `listProviders(capability?)` 统一入口；env-key 映射声明式集中（仿 pneuma `envMapping`）。
- **改写方向**：把 `src/server/providers/registry.ts`（video: runway/sora/kling/seedance）与 TTS（edge/OpenAI）收编进 `src/providers/registry.ts`；`initProviders` 一处装配所有能力；删除平行 registry + `getProvider as getVideoProvider` 别名；更新 CONTEXT.md 把 video/TTS 纳入不变量 #2 描述的单一入口。
- **不变量**：兑现 #2「provider 经 `src/providers/registry.ts` 注册」；仍只走 OpenRouter（不新增直连厂商）。

### 深模块 ③ — Migrations 注册表 + schemaVersion（W7.5，应测试诉求纳入）

- **接口形状**：每个迁移是 `from → to` 的纯函数；`migrate(doc)` 按 `schemaVersion` 链式跑到最新并写回。
- **改写方向**：composition.yaml / carousel.yaml schema 加 optional `schemaVersion`（默认 `1`）；建 `src/shared/migrations/` 注册表骨架；把现有 runtime 内联的 `migrateLegacyTrackIds` 与独立脚本 `migrations/strip-pipeline.ts` 收编为注册表成员。
- **范围克制**：v0.1.1 只立结构 + 收编现有迁移，**不实现完整 boot-time runner、不改已知债 #57 的迁移行为**（保留现状，只为它留好挂点）。不变量 #3（composition.yaml SSoT + zod + atomic write）不被破坏：迁移走同一 zod + atomic write 路径。

### 重构 / 治理（非深模块）

- **W7 api.ts 拆分**：复用已证的 `src/server/bridge/routes.ts` 子 router 模式，按域拆 `src/server/routes/{works,render,generate,audio,trends,analytics,agent}.ts`，各导出子 Hono router 挂载到主 app（`apiRoutes.route(...)`）。纯结构搬移，无契约变更。
- **W7 root 归类**：`config/logger/paths` → `src/infra/`；`work-store/memory/analytics-collector/audio-tools` → `src/domain/`；同步 CONTEXT.md code map。
- **W6 Skill co-location**：`skills/autoviral/manual/` → `manual/_shared/`（命名/单位/viewer 协议/error-codes）+ `manual/{video,carousel}/`；SKILL.md 改成按 `work.type` 分发的索引；新建 `manual/carousel/02-schema.md` 文档化 CarouselSchema；CLI 加 `carousel add-slide`/`set-layer`；recipes 分区 + carousel 范例；更新 system prompt 图文分支从"直接写 carousel.yaml"改为"用 CLI + 查 docs"。不变量 #5（skill 仅工位操作，ADR-003）保持。
- **W2 文档双轨**：`AGENT.md`（薄包装指向 CLAUDE.md 共享部分 + backend 段）；README 重写目录/架构/provider 节到 0.1.0 现状（删 `modules/`/`taste/`/`check_providers.py` 死引用、provider 表收敛为 OpenRouter-only）；`docs/adr/README.md` 索引表 + 状态机；CONTRIBUTING Version Bump Checklist；CONTRIBUTING 集中"运维 Known Gotchas"锚点补 `@remotion/transitions` exact-pin 等高频坑。
- **W1 清债**：`git rm svelte.config.js web/package-lock.json test-studio.mjs ecosystem.config.cjs`；`.gitignore` 加 `.vite/` + `git rm --cached .vite/deps/*`；`tsconfig.json` exclude `*.test.ts`/`__tests__`（或拆 `tsconfig.build.json`）。
- **W3 CI 类型门**：`package.json` 加 `typecheck:web`（`tsc --noEmit -p web/tsconfig.json`）；`ci.yml` 加 typecheck step；修 `ci.yml` 假注释。

## Testing Decisions

**什么是好测试**：只测外部行为（接口契约 / 用户可见结果），不测实现细节；改实现不该改测试。UI 终点以**浏览器截图**为准（不变量 #6 + `.claude/rules/e2e-testing.md`），后端 artifact 不算 E2E 成功。顺序跑、遵守 `maxThreads/maxForks=2`（不变量 #7），绝不并发两个 vitest。

**要写隔离测试的模块**（经与维护者确认）：

1. **ContentTypeRegistry** — 断言两个内容类型注册项的 `deliverableFile`/`routePath`/`seedFactory`/`checkpointTargets` 正确；断言 `DELIVERABLES` 从注册表派生覆盖两类产物；加第三条 mock 注册项验证消费方（NewWorkCard 渲染、WorksGrid 路由）无需改接口即生效。
2. **MediaProvider registry** — 断言 image/video/TTS 三类 provider 经统一 capability 接口可解析、`getDefaultProvider(capability)` 正确、env-key 映射正确；断言旧 `getVideoProvider` 别名删除后无回归。
3. **Migrations 注册表** — 断言无 `schemaVersion` 的旧 yaml 按版本链迁移到最新；断言 `migrateLegacyTrackIds` 收编后行为不变（同输入同输出）；迁移走 zod + atomic write（不变量 #3）。
4. **docs-drift 守卫** — 从 `buildSystemPrompt` 输出用 regex 提取所有 `autoviral docs (NN-slug)` 与 `manual/NN` 引用，断言每个 `fs.existsSync(join(manualDir, topic+'.md'))`；去掉现有 OR fallback 让它真咬合；故意改一个 topic 名能让它变红（守卫有效性自证）。

**Prior art（仓内同类测试范式）**：

- registry/纯函数隔离测试 → 参考 `src/work-store.test.ts`、`src/composition/` 现有 zod schema 测试。
- contract sweep matrix（枚举整个 surface family 而非今天列出的 N 处）→ 参考 `skills/autoviral/contracts/` 思路 + memory `feedback_contract_test_sweep_gate`。
- E2E 浏览器可见 → 参考 `.claude/rules/e2e-testing.md` Hard rules 1–5 + 既有 Studio E2E 流程。
