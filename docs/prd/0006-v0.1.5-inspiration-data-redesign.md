## Problem Statement

我（创作者，当前 5 粉丝、9 件已发布作品）打开 AutoViral 的「数据」和「灵感」两页，得不到任何对创作有帮助的东西：

- **数据页是个空壳**。顶部只有三个孤立数字（平均点赞 16 / 平均评论 0 / 互动率 2.6%），没有任何基线告诉我这是好是坏。年龄分布 / 性别占比 / 热门地域三张卡和「最新调研洞察」全都写着"暂无…等待后台采集首批样本"——像在等一个永远不会来的后台任务。零零散散，对我没有任何帮助。
- **灵感页是被动的趋势浏览器**。最显眼的「起手切角灵感」卡是写死的占位样例，「生成 →」按钮是灰的点不动。趋势列表对所有人都一样，我填在设置里的 穿搭/健身/科技 兴趣过滤不了任何东西。我不能问"这周我该做什么选题"，不能就一个趋势追问下去，没有任何交互感。

我想知道：现在的实现到底差在哪、灵感页是不是该有个能对话的 agent、对标专业产品我缺什么、怎样才能真正满足我的需求。

## Solution

把两页从"空壳 + 被动浏览"重做成"**诚实的数据 + 有根的教练**"。两条铁律贯穿始终：**(1) 绝不展示拿不到的数据、绝不用假承诺骗用户；(2) 把已经拥有却藏起来的真实数据和已经建好却睡着的能力唤醒。**

**数据页**——停止依赖那条已退役的采集管线，改为：
- 把**已在磁盘、已被适配器解析、却从没上屏**的 9 件作品真实 per-post 指标（播放/点赞/评论/分享/收藏）和平均播放（624）渲染成可排序的作品表现表 + 平均播放 KPI；
- 把每个 KPI 放进一条**静态 benchmark 百分位带**里，让"互动率 2.6%"变成"低于 nano 层中位数，目标区间 X–Y"这样的诊断句；
- **删掉**（不是推迟）3 张人口属性卡和洞察空卡——这些是平台 OAuth-only 的私有数据，在 5 粉规模下任何平台 API 都不返回，后台采集架构上不可能合成——换成诚实的三段式空态（说明为什么 / 带水印的示例形状 / 一个真实可行的 CTA）+ 一张"平台诚实矩阵"；
- 重建抖音采集器（**托管 Python venv** 跑 f2 + 浏览器 cookie 授权），让那 9 件作品的数据能刷新而非冻在 5-14；
- 用本地 agent 生成"最新洞察"，但**受诚实护栏约束**：只能引用磁盘上真实存在的指标（播放/点赞/评论），绝不编造 AutoViral 从未测量的留存/完播。

**灵感页**——从被动浏览升级为可对话的创作副驾：
- 在灵感页挂一个**持久的「研究/策略 coach」**对话——复用已建好的 ChatPanel/聊天传输，唤醒那个**已经写好却零前端调用**的研究 agent（`createTrendSession`），读我的 9 件作品 + 趋势 + 兴趣，给打分过的选题，一键落成新作品；
- 用真正个性化的选题 brief 替掉写死的假「起手切角」卡；
- 给趋势加 drill-down + 新鲜度/来源诚实角标，并把我声明的兴趣真正喂进趋势排序。

## User Stories

**数据页 · 把已有真实数据接上线**
1. As a 创作者, I want 看到我每件已发布作品的真实播放/点赞/评论/分享/收藏, so that 我知道哪件表现好、哪件沉了，而不是只看一个聚合平均数。
2. As a 创作者, I want 一个"平均播放"KPI 和可按播放/点赞排序的作品表, so that 我能一眼找出我的爆款和哑炮。
3. As a 创作者, I want 每个 KPI 旁边有一条同层级创作者的基线带, so that "互动率 2.6%"对我有意义——我知道这是高于还是低于我这个粉丝量级的中位数。
4. As a 创作者, I want 系统在我互动率偏低时附一句"小账号天然互动更高，别灰心", so that 基线不会打击我而是引导我。
5. As a 创作者（已发布作品但数据稀疏）, I want 把我这几件作品和我自己的历史比, so that 我能看到自己在变好还是变差，而不是被拿去跟头部比。
6. As a 创作者, I want 一张成长轨迹 + 下一个里程碑卡（如"从 5 到 50 关注者"）, so that 在数据点很少时我仍有一个向前看的目标。〔Wave 2〕
7. As a 创作者, I want 我的 9 件作品被打上 3–5 个内容支柱标签并比较各支柱表现, so that 我知道"我的 X 类作品互动是 Y 类的 2 倍"。〔Wave 2〕

**数据页 · 诚实地对待拿不到的数据**
8. As a 创作者, I want 数据页**不再**显示我永远填不满的年龄/性别/地域卡, so that 我不会一直盯着假装在加载的空卡。
9. As a 创作者, I want 任何空态都告诉我**为什么**空、**何时**能有、以及**一个**我现在能做的动作, so that 空态是有用的指引而不是死胡同。
10. As a 创作者, I want 一张"平台诚实矩阵"清楚列出每个平台{能否拿自有数据 / 能否拿人口属性·门槛多少 / 趋势数据是实采还是 LLM 推理}, so that 我对这个工具能给我什么有真实预期。
11. As a 创作者, I want 看到每个数字的来源和最后同步时间, so that 我不会把一个月前的旧数当成实时。

**数据页 · 让数据能刷新**
12. As a 创作者, I want 用我浏览器里已登录的抖音身份把我作品的最新数据抓回来, so that 数据不再冻结在某个历史快照。
13. As a 创作者, I want 采集器作为托管依赖被自动装好, so that 我不用手动配 Python 环境。
14. As a 创作者, I want 采集失败（cookie 过期 / 改版）时得到清楚的、可操作的提示, so that 我知道该重新登录而不是面对一个静默的空页。
15. As a 创作者, I want "最新洞察"由本地 agent 读我的真实作品生成, so that 我能拿到定性建议（如"你某类作品的钩子更留人"）。〔Wave 2〕
16. As a 创作者, I want 这些 agent 洞察绝不编造它没测过的指标（留存/完播）, so that 我能信任每一句洞察都有数据支撑。〔Wave 2〕

**灵感页 · 有根的对话教练**
17. As a 创作者, I want 灵感页上有个能对话的策略教练, so that 我能直接问"基于我已发布的 9 件作品，下一个该做什么选题"。
18. As a 创作者, I want 这个教练读得到我的真实作品 + 当前趋势 + 我的兴趣, so that 它的建议是针对我的、不是泛泛而谈。
19. As a 创作者, I want 空白对话框里有一排起手提问（prompt library）, so that 我不用面对空白框发懵。
20. As a 创作者, I want 教练把每个选题打分并解释为什么值得做, so that 我能挑最有把握的去做。
21. As a 创作者, I want 一键把教练给的某个选题落成一件新作品, so that 从灵感到动手之间没有摩擦。
22. As a 创作者, I want 这个对话能跨轮、跨重载保留历史, so that 它像一个持续了解我的教练，而不是每次从零开始。
23. As a 创作者, I want 教练对薄数据保持诚实（数据太少就说太少、转而靠趋势/兴趣），so that 它不会假装精确。
24. As a 创作者, I want 教练有 token 成本上限、不会每轮重读我全部作品, so that 它不会烧光我的 API 额度。

**灵感页 · 趋势真正服务于我**
25. As a 创作者, I want 删掉那张写死的假「起手切角」卡, so that 最显眼的位置不再是个点不动的占位符。
26. As a 创作者, I want 顶部是 agent 基于我作品+趋势+兴趣生成的真实个性化选题 brief（推送式）, so that 我一上来就有"为你的赛道准备的 5 个选题"。
27. As a 创作者, I want 点开一个趋势看到趋势线、可看的样例（来源允许时）、相关切角, so that 我能从抽象趋势走到具体能做的选题。〔Wave 2〕
28. As a 创作者, I want 趋势带"上升/爆发"紧迫角标（如"建议 72h 内发布"）, so that 我知道哪些趋势要趁早。〔Wave 2〕
29. As a 创作者, I want 我设置里填的兴趣真正影响趋势排序, so that 趋势是按"与我赛道的契合度"排的、不是对所有人一样。〔Wave 2〕
30. As a 创作者, I want 每个趋势诚实标注它是实采还是 LLM 推理 + 多久前采的, so that 我不会把 AI 编的话题当成真实平台情报。〔Wave 2〕

## Implementation Decisions

> 架构精神沿用 ADR-009：**把"诚实"与"正确性"做成可隔离测试的纯函数核心（deep module），UI 只是渲染薄壳**。下列 D1–D5 是 deep module，S1–S8 是 wiring。不写具体文件路径（易过时），只定接口与决策。

### 模块（deep，全部写 TDD 测试）

- **D1 `creator-analytics` 派生核心**。输入采集器/快照里的 `works[]` + `summary`，纯函数派生：平均播放、每件作品表现行（可排序）、自我历史对比、内容支柱聚合、成长轨迹/下一里程碑。**关键纠正**（completeness-critic 实证）：这些 per-post 指标**已在磁盘**（冻结的 `latest.json` 含全部 9 件作品的 play/digg/comment/share/collect + `avg_play=624`），且适配器**已解析** `works[]`/`avgPlay`，只是 UI 从没渲染——所以这是 **built-not-wired 的 S/M 接线活，不是新采集**。冻结的 `latest.json` 即天然测试 fixture。

- **D2 `benchmark` 百分位带**。接口：`(platform, 粉丝层, metric, value) → {带位置, 诊断文案, 安抚文案}`；基线数据是**仓库内静态 JSON**（per 平台 × 粉丝层），零实时采集。**关键约束**（critique 实证）：外研引用的多是 IG/TikTok/YT 基线；**必须为用户真实平台（抖音）配 platform-correct 带，否则就近用别平台的带是"伪诊断"、反而破坏诚实**。若某平台无可信公开基线，则明确标注"参考性、非你所在平台"。

- **D3 `insight-guardrail` 诚实护栏**。接口：`(候选洞察[], 磁盘可用metric集) → 通过的洞察[]`；任何引用了不在可用 metric 集里的指标（尤其 retention/完播/钩子留存——AutoViral 从未测量）的洞察一律拒绝。这是把"诚实"从人肉 review 升级为**测试门控**的核心。〔服务 Wave 2 的 S4，但护栏纯函数本身在 Wave 1 就可建可测。〕

- **D4 `douyin-collector`（托管 venv）**。**用户决策：托管 Python venv，不纯 TS 重写**——因为难点是抖音 `a_bogus`/`X-Bogus` 请求签名，f2 库在 Python 侧已解决，纯 TS 重写等于自己追着抖音改版维护签名。恢复 f2 + `browser_cookie3` 采集器作为**托管依赖**，跑在 `~/.autoviral` 的托管 venv（**复用 v0.1.2 给 TTS 装 edge-tts 的同一套 doctor / managed-bin / venv 引导机制**）。接口：`refresh(profileUrl) → CreatorData | CollectorError`；解析边界（raw f2 JSON → `CreatorData`）是纯函数、可 fixture 测试，实际抓取是集成。把 `POST /api/analytics/refresh` 从硬 501 改为真刷新。账号识别：用户配置抖音 profile URL → 解析 `sec_user_id`。鉴权：读用户已登录浏览器的 `sessionid` cookie（**首跑需用户在 douyin.com 登录授权 = 本 PRD 唯一 HITL 触点**）。

- **D5 `coach-session`（持久「研究/策略」persona）**。**用户决策：持久对话**。在聊天传输层新增一个**带 sidecar 持久化**的 session kind（区别于现有故意无历史的 ephemeral `trends_` 会话）。新增 `buildCoachSystemPrompt`：persona = 研究/策略教练（非现有的剪辑/交付 persona），上下文 = 读 9 件本地作品 + 选定平台趋势 artifact + config 兴趣/嗓音。**session 级 model 作用域**——修掉现有 ModelSwitcher 改全局 `config.model` 的问题，避免灵感 coach 与剪辑 agent 抢同一档位。成本护栏：每会话 token 预算 + 惰性上下文加载（不每轮重读全部作品）+ 只读/建议契约。

### 决策（wiring 与架构）

- **删除而非推迟人口属性卡**（critique 纠正 synthesis 的"defer YouTube"软化）：年龄/性别/地域是 owner-OAuth-only 私有数据；用户真实平台里**小红书无个人 API、抖音需企业主体**，且 5 粉规模下任何平台都不返回——所以诚实的唯一做法是**删卡**，并在平台诚实矩阵里讲清楚。YouTube connect-channel OAuth 真人口属性 = **明确 Out of Scope（Wave 3）**，不在本版承诺。
- **诚实三段式空态**作为可复用组件：Inform（为什么空）/ Inspire（带水印的示例形状，绝不与真实数据混淆）/ Activate（一个真实可行 CTA；区分"连接账号"vs"粉丝还不够"vs"该平台无 API"）。删掉指向 501 的"刷新"死胡同 CTA。
- **解耦 ChatPanel ↔ workId**：现有 ChatPanel 的历史 seed / 附件上传目标 / checkpoint 回滚 / viewer-context 都硬绑 workId；coach 是 workless（或独立持久 session），需把这些解耦——这是 S5 的真正工作量所在。
- **一键选题→作品**复用 #65 `buildTrendTopicHint` 管线，但接的是 **chat 输出**而非趋势行（是一个不同的发起面，需新接线）。
- **趋势新鲜度/来源诚实**：`GET /api/trends/:platform` 当前不论数据多旧都当当前返回（前端 404 时甚至把 collectedAt 默认成 now）；加 last-collected 时间戳 + stale 角标 + 来源 provenance。〔与 0005 的 B2/B6 同源，本 PRD 的 S7/S8 覆盖其 trends-route 部分。〕
- **兴趣感知趋势排序**把 `config.interests` 喂进采集/富化路径（当前 grep 不到 interests 被消费）。**门控**（critique）：4 平台里 3 个是 LLM 编造（`agent_websearch`、metrics 全 null），给 vapor 排序无意义——S8 必须**先诚实标注来源**，排序质量受底层数据真实度上限约束。

## Testing Decisions

- **只测外部行为，不测实现细节**。好的测试断言模块的输入→输出契约（如"给定这份 works fixture，派生出的平均播放=624、最高播放作品=X"），而非内部函数怎么写。
- **5 个 deep module 全部 TDD（红→绿，每片先写失败测试）**——用户决策。具体：
  - **D1**：以冻结的 `latest.json` 为 fixture，断言平均播放、排序、自我对比、支柱聚合、轨迹派生正确。
  - **D2**：断言带位置/诊断文案正确；**专门断言抖音的带是 platform-correct 或被明确标注为参考性**（防"伪诊断"回归）。
  - **D3**：断言任何引用 retention/完播 等不在可用 metric 集的洞察被拒；只引用 play/digg/comment 的通过。这是诚实性的回归门。
  - **D4**：以一份 raw f2 JSON 为 fixture，断言解析边界产出正确 `CreatorData`；cookie 过期/改版 → 结构化 `CollectorError`。实际抓取走集成、不进单测门。
  - **D5**：断言 coach session kind 走持久 sidecar（非 ephemeral 路径）、persona prompt 含本地作品/趋势/兴趣上下文、model 作用域是 session 级而非全局。
- **UI wiring（S1–S8）靠 Workflow 多纬度浏览器 E2E 验收**（per e2e-testing.md Hard rule 0），主 agent 不自己点浏览器；不写脆的 UI 单测，关键交互（coach 挂载、空态、作品表渲染）以 E2E 截图 + DOM 二确为准。
- **Prior art**：沿用仓库现有 vitest 套件范式（web `npm run test:web` 一次性跑、server `npm run test:server`；`maxThreads/maxForks=2` 上限不可移除，见 CLAUDE.md `<testing>`）。nodenext 下测试 import 记得带 `.js` 扩展（见 peaks 测试教训）。

## Out of Scope

- **YouTube/IG connect-channel OAuth 真人口属性**（Wave 3 / 未来版本）：是拿真实年龄/性别/地域的唯一诚实路，但用户主要不在 YouTube、5 粉也无返回、需 Google 验证审核——对当前零价值，本版只把人口卡**删掉**，不接 OAuth。
- **用户真实平台（抖音/小红书）的真人口属性**：架构上不可得（小红书无个人 API、抖音需企业主体、规模门槛），本版不承诺、不假装。
- **让 YouTube/TikTok 趋势源变真**：仍是 `agentFallback`（LLM 推理），本版只**诚实标注**来源，不实现真采集。
- **合并两条互相打架的趋势管线**（`researchTrends` vs `createTrendSession`，同目录不同 schema 靠 normalize shim 缝合）：记为技术债，本版只唤醒 createTrendSession 的前端，不做管线统一。
