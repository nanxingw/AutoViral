## Problem Statement

AutoViral 已经走完 agentic-terminal 重构（6 个 phase 全部打 tag、terminal/bridge/17-cmd CLI/skill manual 均为真实代码），但项目还没有一次**正式的、可分发的发布**，并且工程治理层留下了一批互相牵连的债，使得「发一个版本」这个动作目前是不可重复、不可验证的：

1. **包身份与决策脱节**。根包 `package.json` 仍是 `@nanxingw/autocode-cli@0.2.0`，`bin` 同时导出 `autocode` 与 `autoviral` 两个名字指向同一个 `dist/index.js`，`repository.url` 指向 `github.com/nanxingw/autocode`，description/keywords 还在讲「AutoCode self-evolving intelligence」。CLI 运行时（`src/cli.ts`）的 `.name()` / `.version()` 与 8 处 `postinstall` 日志前缀也都还叫 `autocode`。直接 publish 会以错误的名字、错误的 scope、错误的版本号上架。

2. **两个 `autoviral` CLI 在 bin 名上撞车**。根包与 `cli/autoviral`（`@autoviral/cli@0.1.0`，Studio 内 agent 调用的协议层 CLI）都声明了名为 `autoviral` 的 bin。全局同时安装两者会互相覆盖；而 `cli/autoviral` 根本不在根包的 `files` 白名单里，干净机器上 `npm i -g` 装到的是 daemon，agent 侧的 `autoviral whoami` 会 command-not-found。

3. **docs 结构无规范**。`docs/` 下平铺着 `superpowers/`、`agents/`、`adr/`、`qa/`、`desigen/`（拼写错误）、研究 md、截图等，没有统一入口，PRD 也没有固定家。`docs/superpowers/` 这个名字与项目当前的 skill 体系（已按 ADR-004 迁移到 mattpocock）名实不符——它现在装的是 61 份 plans/specs/notes，本质是历史归档，不是 superpowers skill。

4. **发布不可重复**。根包没有 `prepublishOnly` / `prepack` 守卫，tarball 直接 ship 工作区里碰巧存在的 `dist/`（而 `dist/` 是 gitignored）；`files: ["dist/"]` 还把 186 个 `dist/**/*.test.js` 与 193 个 `*.js.map` 一起 ship 出去，泄露测试逻辑、撑大体积。没有桌面壳、没有 CI、没有 release 自动化。

5. **没有桌面分发形态**。产品是一个常驻 daemon（`node dist/index.js start`，端口 3271）+ Vite/React SPA + Remotion 渲染 + 系统 ffmpeg + python venv 的组合，对开发者友好，对终端用户「双击即用」不友好——尤其 ffmpeg 不在 PATH 是 MEMORY 里反复记录的头号首次运行故障。

这五条互相咬合：不统一包身份就不能 publish；不解决 bin 撞车就会装坏；不规范 docs 就没有 PRD 的家也没法在发布说明里引用历史决策；不加发布守卫与 CI 就无法保证「发出去的就是 main 上的、且测试是绿的」；没有桌面壳就只有命令行形态。本 PRD 把它们作为**一次性整桩**解决，产出 `v0.1.0`。

## Solution

把 `v0.1.0` 定义成「AutoViral 第一次可重复、可验证、可分发」的发布，一次性落地以下七件事：

1. **版本与命名约定**：项目采用 semver，公开制品命名为 `autoviral`，本次版本 `0.1.0`（pre-1.0），包管理器统一 `npm`。
2. **npm 打包为 `autoviral@0.1.0`**：根包改名、降版本、收敛 bin、修 repository/description、加发布守卫、剔除 test/map 泄漏。
3. **docs 重组**：`docs/superpowers/` → `docs/archive/`（完整保留全部 61 份 plans/specs/notes），并建立 `docs/{README.md, prd/, adr/, agents/, reference/, design/, qa/, archive/}` 标准结构；新增 `docs/prd/`（本文件即首份）。
4. **CLAUDE.md 更新**：把指向 `docs/superpowers/` 的进度/协议链接改到 `docs/archive/`，记录新的 PRD/版本约定。
5. **Electron 桌面壳**：thin-host 模式包裹现有 daemon，bundle ffmpeg，零前端改动。
6. **GitHub Actions**：CI（每次 push/PR 跑 build + 双 vitest 套件）+ Release（semver tag 触发，矩阵构建桌面安装包并 publish CLI）。
7. **合并 `refactor/agentic-terminal` → `main`**：以 `v0.1.0` 落地。

核心原则（遵循 CLAUDE.md「Boil the ocean」）：能一次性永久解决就一次性解决，不留「以后再做」。所有「完成」断言都必须有可观测验证（build 通过、vitest 全绿、`npm pack --dry-run`、`electron-builder` dry run）背书，详见 ## Testing Decisions。

## Goals & Non-Goals

### Goals

- G1：根包以正确身份 `autoviral@0.1.0` 通过 `npm pack --dry-run`，tarball 不含 `*.test.js` / `*.js.map`，且有 `prepublishOnly` 保证 ship 的是新鲜 build。
- G2：`docs/` 落地标准结构，`docs/superpowers/` 物理迁移到 `docs/archive/`，全仓零残留旧路径引用（`rg -i "docs/superpowers"` 为 0），61 份历史文档一份不少。
- G3：Electron 桌面壳可在本机 `desktop:dev` 跑起来、`desktop:build` 产出 dmg/nsis；ffmpeg + ffprobe 随包，首次运行无需用户装 ffmpeg。
- G4：`.github/workflows/ci.yml` 在每次 push/PR 跑 `npm run build` + `test:web` + `test:server`（尊重 maxThreads/maxForks=2 内存上限）；`release.yml` 在 `v*.*.*` tag 上构建桌面安装包并 publish CLI。
- G5：`refactor/agentic-terminal` 合入 `main`，打 tag `v0.1.0`，CHANGELOG 有对应条目。
- G6：bin 名撞车有明确裁决，两个 CLI 的分发关系被显式定义且文档化。

### Non-Goals

- N1：不做代码签名 / 公证（mac notarization、win Authenticode）——0.1.0 接受 unsigned，见 ## Out of Scope。
- N2：不引入 per-component 独立版本号——0.1.0 单一版本号驱动整个仓。
- N3：不做 Linux 桌面 target（0.1.0 只 mac + win）。
- N4：不重写前端、不改 bridge 协议、不改 Remotion 渲染管线——桌面壳是 thin host，零前端改动。
- N5：不解决 python venv（TTS/peaks）的 bundle——那是 ffmpeg 之外的独立问题，0.1.0 仍假设系统 python3。
- N6：不引入 changesets / semantic-release / standard-version 等重型 release 框架——保持手动 semver bump + 手写 CHANGELOG + 轻量 workflow。

## User Stories

从三类使用者视角：

1. **作为一个刚 onboard 的新 dev agent**，我打开仓库时应能在 `docs/README.md` 找到统一入口，知道 PRD 在 `docs/prd/`、决策在 `docs/adr/`、历史 plans/specs 在 `docs/archive/`，这样我无需翻 git log 就能定位「现在在建什么、为什么这么建」。

2. **作为新 dev agent**，当我读到任何文档或源码注释里指向 `docs/superpowers/...` 的链接时，它都应该已经被改成 `docs/archive/...` 并真的能点开，这样我不会因为死链而去重建别人已经写好的方案（对抗 MEMORY 里反复出现的「grep existing helper first」教训）。

3. **作为一个要 cut release 的 maintainer**，我希望 push 一个 `v0.1.0` tag 就能自动产出 mac/win 安装包并把 CLI publish 到 npm，且发布前 CI 已经验证 build 绿、双 vitest 套件绿，这样 release「永远不会带红上车」。

4. **作为 maintainer**，我希望 `npm publish` 出去的就是 `autoviral@0.1.0` 这个正确身份，bin 不撞车、tarball 不夹带 test/map、`dist/` 一定是当下 build 的结果，这样不会把错误身份或陈旧产物推上公共 registry。

5. **作为 maintainer**，我需要一份明确的 semver 与 CHANGELOG 约定，知道什么改动 bump major/minor/patch、release notes 从哪里来，这样版本号与变更记录不会随手而来、互相漂移。

6. **作为一个下载桌面 app 的终端用户**，我双击 AutoViral.app（或运行 nsis 安装包）就能用，不需要单独装 Node、不需要 `brew install ffmpeg`、不需要懂端口 3271，渲染和波形开箱即用。

7. **作为终端用户**，0.1.0 是 unsigned 的，我第一次启动会遇到 Gatekeeper / SmartScreen 拦截，但发布说明里有一行明确的绕过指引（mac 右键打开或 `xattr -dr com.apple.quarantine`，win「更多信息 → 仍要运行」），不会让我卡在「无法打开」上。

## Implementation Decisions

### 1. 四个锁定决策

| 维度 | 决策 | 说明 |
|---|---|---|
| 包名 | `autoviral`（unscoped） | 取代 `@nanxingw/autocode-cli`，干净的新身份 |
| 版本 | `0.1.0` | 从 0.2.0 降到 0.1.0：这是一次破坏性的重命名重启，pre-1.0 语义见下 |
| 包管理器 | `npm` | 全仓统一 `npm ci` / `npm run`，不引入 bun/pnpm |
| 桌面 | `electron` + `electron-builder` | thin host 包裹现有 daemon，非重写 |

### 2. docs 结构与 superpowers 归档

`docs/` 标准结构（这是规范本身，prose 描述不如直接给结构块）：

```
docs/
  README.md      # 统一入口：指向下面每个目录的用途
  prd/           # 产品/治理 PRD（本文件 = 0001），NNNN-kebab-title.md
  adr/           # 架构决策记录 ADR-NNN-*.md（保持现有 5 份不动）
  agents/        # agent 协作约定（issue-tracker / triage / domain / index）
  reference/     # 长青技术参考
  design/        # 设计稿与视觉规范（吸收原 desigen/ 拼写错误目录）
  qa/            # 测试/验收记录
  archive/       # 历史 plans/specs/notes（原 docs/superpowers/ 整体迁入）
    plans/       # 原 superpowers/plans/
    specs/       # 原 superpowers/specs/
    notes/       # 原 superpowers/notes/
```

迁移方式：`git mv docs/superpowers/ docs/archive/`，完整保留全部 61 份文件（plans + specs + notes），文件内部的相对交叉引用随目录改名自动解析，无需逐文件改内容。**只**改「指向 `docs/superpowers/` 的外部路径引用」——共 13 处分布在 11 个 doc + 5 个源码注释 + 1 个脚本（与下面 step 一致），全部是机械字符串替换 `docs/superpowers/` → `docs/archive/`，无逻辑改动。

**不改**的引用（历史决策上下文，名实仍准确）：`ADR-004-mattpocock-replaces-superpowers.md`（主决策文档，Accepted）整篇保留；`ADR-003` 的迁移叙述、`CLAUDE.md` 第 8 行「This replaces `superpowers:*`」、`src/ws-bridge.ts` 里「不要用 `superpowers:*`」的 guardrail、`ws-bridge.test.ts` 里的 `"superpowers:brainstorming"` 断言——这些讲的是「我们用 mattpocock 取代了 superpowers skill」这件事本身，与目录改名无关，改了反而失真。

### 3. 包身份与 bin 名裁决

根包 7 个身份标记必须**原子地**一起改（否则用户拿到错误身份）：`name` → `autoviral`、`version` → `0.1.0`、`bin`、`repository.url` → AutoViral 仓、`description`/`keywords` → AutoViral。运行时一致性同改：`src/cli.ts` 的 `.name()`/`.version()` 与 3 处面向用户的 `autocode` 文案、`src/postinstall.ts` 的 8 处 `autocode:` 日志前缀。`package-lock.json` 不手改，`npm install` 后自动同步。

**bin 名裁决（锁定）**：根包 `bin` **只导出 `autoviral`，彻底删掉 `autocode`**。理由：双 bin 指向同一个 `dist/index.js` 只制造混淆，且延续了已死的 AutoCode 品牌；删掉是干净地宣告「AutoViral 是新身份」。若将来真需要向后兼容旧 `autocode` 命令，用安装期 symlink/shim 解决，不再以 published npm bin 形式存在。

### 4. 两个 CLI 的关系与撞车解决

仓内有两个独立可发布的包，它们**不是为联合分发而绑定的**：

- **(A) 根包 = daemon/Studio 入口**：改名后为 `autoviral`，bin `autoviral` → `dist/index.js`，即 `node dist/index.js start`、端口 3271 的常驻 daemon。
- **(B) `cli/autoviral` = `@autoviral/cli@0.1.0`**：Studio 内被 spawn 的创作 agent 调用的协议层 CLI，bin 也是 `autoviral` → `./dist/cli.js`。它本就不在根包 `files` 里，靠本地 `npm link`（`install:cli`）使用，其 `dist/` 是 gitignored。

两者 bin 名同为 `autoviral` 会在全局安装时撞车。**0.1.0 的裁决**：这是**两个不同的分发制品**，按用途分别走不同通道——根 daemon 走桌面安装包（不依赖全局 `autoviral` bin，Electron 直接用 `process.execPath` 跑 `dist/index.js`，见下）；`@autoviral/cli` 走 npm publish（Release workflow 的 Job B，从 `cli/autoviral` 目录 publish）。桌面 app 内 agent 的 `autoviral` 命令由 spawn-env PATH/shim（commit 25fd7a0 已落地）解析，不依赖全局安装。因此「全局同时装两者」这一冲突场景在分发路径里被结构性绕开。本决策必须在 tag 前文档化（见 ## Open Questions Q1）。

### 5. ffmpeg 策略

**决策：用 `ffmpeg-static` bundle ffmpeg，不要求系统 ffmpeg。** 桌面 app 必须「双击即用」，而 ffmpeg 不在 PATH 是 MEMORY 反复记录的头号首次运行故障，bundle 一个测试过的静态二进制同时消除「装不上」与「版本漂移」（loudnorm/xfade/image2pipe 散落在 audio-tools / transitions / streaming-encoder / speed-ramp / render-pipeline 多处 codepath）。

**load-bearing 注意点**：daemon 以 **bare name** spawn ffmpeg/ffprobe（6+ 处 `spawn("ffmpeg")`、`execFile("ffprobe")`），只走 PATH 解析、今天不读 `FFMPEG_PATH` 环境变量。0.1.0 走「零 daemon 改动」方案：Electron 主进程把 unpacked 的 ffmpeg 目录**前置**到子 daemon 的 PATH。由于 `ffmpeg-static` 只带 `ffmpeg`（没有 `ffprobe`），**必须额外 vendor 一个静态 ffprobe** 进同一目录。「让 daemon 读 `FFMPEG_PATH`/`FFPROBE_PATH` 并回退 bare name」是更干净的 fast-follow，但它改 shared daemon 源码，0.1.0 ship 方案一并开 issue 跟踪方案二。

**跨架构**：`ffmpeg-static` 只装构建宿主架构的二进制。0.1.0 务实路径是 GitHub Actions 矩阵——`macos-latest` 原生出 arm64 dmg、`windows-latest` 原生出 x64 nsis，各 runner 的 `ffmpeg-static` 天然给对架构的二进制。

### 6. Electron thin-host 形态

桌面壳是新的顶层 `desktop/` 目录，加进**现有根包**（不开第二个包——daemon、web/dist、skills/ 都在根，必须一起 ship）。主进程做三件事：(1) 用 Electron 内嵌 Node（`ELECTRON_RUN_AS_NODE=1` + `process.execPath`，用户无需装 Node）spawn 现有编译好的 daemon（强制 `--foreground`，由壳掌控生命周期，不走 daemonize+PID 那条路）；(2) health-check 轮询 `http://127.0.0.1:3271/` 直到响应（250ms × ~40，10s 预算），不用固定 sleep；(3) 打开一个指向该 origin 的 `BrowserWindow`。renderer 是未改动的 SPA 走 loopback 连自己的 daemon，WS upgrade（terminal/bridge/render）因 same-origin same-port 天然可用。

关键环境变量：`AUTOVIRAL_DATA_DIR` 指向 `app.getPath('userData')/autoviral`（per-user 可写、升级安全，**不**在只读的 `.app` 内）；PATH 前置 bundled ffmpeg 目录。生命周期：`requestSingleInstanceLock` 防两实例抢 3271；退出时 SIGTERM → 等 4s → SIGKILL 杀子进程，杜绝孤儿 daemon 占端口。`electron-updater` + `electron-log` 必须放在 `dependencies`（不是 devDependencies），否则 electron-builder prune 掉、打包后主进程启动即 crash。

### 7. tarball 卫生

`files: ["dist/"]` 会无差别 ship 出 186 个 `*.test.js` + 193 个 `*.js.map`。0.1.0 通过排除 test 与 map（独立 build tsconfig 或 `files` 负向 glob）把 796 文件 / 14.4MB 的 payload 收瘦到只含运行时 `dist/`、`skills/autoviral`、`web/dist`、README。`.env` 已被 `files` 白名单挡在外（保持白名单模式，不切换成 `.npmignore` 黑名单），另加一条 `.env*`/`*.key`/`*.pem` 的 belt-and-suspenders 与 `npm pack --dry-run` grep 作为 CI publish gate。

## Release & Versioning Policy

- **版本方案**：严格 semver `MAJOR.MINOR.PATCH`。整仓单一版本号（见 Non-Goal N2），不做 per-component 版本。
- **bump 规则**：
  - **MAJOR**：破坏性变更——bridge 协议不兼容改动、CLI 命令/契约移除或改名、composition schema 不向后兼容、桌面数据目录布局破坏。
  - **MINOR**：向后兼容的新能力——新 Studio feature、新 CLI 子命令、新 skill recipe、新桌面 target。
  - **PATCH**：向后兼容的修复——bug fix、文案/i18n、性能、纯 docs。
- **pre-1.0 caveat**：0.x 阶段 API 视为尚未稳定，`0.MINOR` 的提升可能携带行为变化；在 1.0 前不对外承诺 semver 的破坏性保证。把 `0.2.0`（旧 AutoCode fork）重置到 `0.1.0` 正是因为这是一次破坏性的身份重启，pre-1.0 允许这样做。
- **CHANGELOG**：采用 [Keep a Changelog] 格式，标题语法严格为 `## [MAJOR.MINOR.PATCH] - YYYY-MM-DD`，分类 Added / Fixed / Changed / Improved。CHANGELOG 条目即发布说明的单一事实来源——保持标题语法可被 awk 抽取（参考 pneuma：release 流程 awk 提取匹配 `## [version]` 段落喂给 `gh release create --notes-file`，让 CHANGELOG 与 GitHub Release 不会漂移）。
- **tag 约定**：`vMAJOR.MINOR.PATCH`（如 `v0.1.0`）。Release workflow 由 `v*.*.*` tag push 触发。
- **release 工具**：保持轻量——手动 semver bump（package.json）+ 手写 CHANGELOG 条目 + 用 `gh` CLI 创建 Release，不引入 changesets/semantic-release（Non-Goal N6）。tag 必须与 `package.json` version 对齐（见 ## Open Questions Q4，0.1.0 靠纪律，后续可加 version-match guard）。

## Migration Plan

按依赖顺序执行，每步带验证 gate（gate 不过不进下一步）：

1. **建 docs 骨架 + 落本 PRD**。创建 `docs/{prd,reference,design}/`（`adr`/`agents`/`qa` 已存在），写 `docs/README.md` 入口，落本文件。
   *Gate*：`ls docs/prd/0001-v0.1.0-release-and-conventions.md` 存在。

2. **物理迁移归档**：`git mv docs/superpowers/ docs/archive/`。
   *Gate*：`find docs/archive -type f | wc -l` == 61；`docs/superpowers` 不再存在。

3. **重写 13 处旧路径引用**（11 doc + 5 源码注释 + 1 脚本，含 CLAUDE.md / CONTEXT.md / CHANGELOG.md / ADR-001 / ADR-005 / agents 三件 / cli README / skill manual / api.ts / routes.test.ts / schemas.ts / routes.ts / check-d3-words.sh），全部 `docs/superpowers/` → `docs/archive/`，相对链接同理（`../superpowers/` → `../archive/`）。

## Testing Decisions

发布是否「sound」以可观测证据为准，不靠「应该没事」：

1. **build 通过**：`npm run build`（`build:backend` tsc + `build:frontend` vite）零错误退出。这同时是 `prepublishOnly` 的内容，保证 ship 的是新鲜产物。
2. **vitest 全绿（遵守内存上限）**：`npm run test:web`（happy-dom，`maxThreads=2`）与 `npm run test:server`（node forks，`maxForks=2`）**顺序**跑、都绿。绝不并发两个 vitest，绝不绕过 `npm run` 直调 vitest（否则 pretest pkill 钩子失效）。自检：`pgrep -f vitest | wc -l` ≤ 3。CI 里 ffmpeg/python/Remotion 全 mock，故 CI 不装 ffmpeg/python——这是有意的，别「以防万一」加成死重。
3. **`npm publish --dry-run` / `npm pack --dry-run`**：核对 tarball 身份是 `autoviral@0.1.0`、不含 `*.test.js`/`*.js.map`/`.env`、文件清单只有运行时必需项。
4. **`electron-builder` dry / 本机构建**：`desktop:dev` 起窗连 daemon、`desktop:build:mac`（或 `--dir`）产出可启动的 app。打包后必须实际启动一次，验证 native module（better-sqlite3 / node-pty / @remotion compositor）在**打包后**而非仅 `desktop:dev` 下能 load，且 ffmpeg+ffprobe 都解析到 bundled 版本（跑一次真实渲染/波形）。
5. **路径残留 gate**：步骤 3 的 `rg -i "docs/superpowers"` 为 0 是 docs 迁移的硬验收。
6. **E2E（浏览器可见为准）**：桌面壳起来后，按 `.claude/rules/e2e-testing.md`，截图证明 Studio 真的在窗口里渲染、能跑一次最小创作流——backend artifact 不算 E2E 成功，用户视角是 source of truth。

## 实现结果（Implementation Outcome）

> 2026-06-02 收尾。本节记录实际落地状态，以可观测证据为准，诚实标注唯一的本地残留项。

**七条工作流全部落地。** 包身份重置为 `autoviral@0.1.0`、docs 迁移到 `docs/archive/`、CLAUDE.md 链接更新、tarball 卫生、Electron 桌面壳、GitHub Actions（`ci.yml` + `release.yml`）、合并准备均已完成。

**daemon 级 packaged-resources 渲染 E2E 通过（浏览器可见为准）。** 用打包态环境变量（bundled ffmpeg/ffprobe/Chromium/预构建 Remotion bundle）拉起 daemon 后，在 Studio 里真实导出了一段视频：RENDER 阶段走的是 bundled Chrome Headless Shell，ENCODE 阶段走的是 bundled ffmpeg，产出一段合法的 h264/aac 1280x720 24fps **78.6s mp4**（由 bundled ffprobe 与 Studio UI「Export complete 100%」双重确认）。这不是 backend artifact 包装——是用户视角的可见导出。

**五个已验证的 CRITICAL 全部解决：**

- **C1 — `dist/package.json` 的 `type: module` unpacked**：asar 内显式 ship `dist/package.json` 并 unpack，ESM 模块解析在打包态下成立。
- **C2 — 预构建 Remotion bundle**：经 `AUTOVIRAL_REMOTION_BUNDLE` 指向随包 bundle，渲染时不再触发运行时 webpack（只读 asar 下无法写 bundle）。
- **C3 — bundled Chromium**：经 `AUTOVIRAL_CHROMIUM_PATH` 指向随包 Chrome Headless Shell，避免运行时向只读 asar 下载 Chromium。
- **C4 — `@electron/rebuild` 重编 Electron-ABI**：electron-builder beforeBuild 钩子按 Electron ABI 重编 better-sqlite3 / node-pty 等原生模块，避免打包后 load 即 crash。
- **C5 — 自包含 `cli/autoviral` + login-shell PATH**：`cli/autoviral` 随包 ship，主进程 login-shell PATH 恢复让 Studio 内 agent 的 `claude` / `autoviral` 命令可解析。

**唯一本地残留 caveat（诚实标注）：** electron-builder 的 `.app` 制品在**本 sandbox** 内无法完整产出——node-gyp 检测不到 better-sqlite3 Electron-ABI 重编所需的 Command Line Tools receipt。这是环境问题（`sudo xcode-select --install`，或在自带 receipt 的 CI macOS runner 上即可解决），**不是配置/代码缺陷**。因此实际安装包由 `release.yml` 在 CI macOS runner 上产出，目前尚未在本地产出完整 `.app`。
