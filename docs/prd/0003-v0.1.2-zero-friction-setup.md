## §1 · 外部依赖自举（zero-friction dependency setup）

### Why / Problem Statement

AutoViral 的核心功能链路**默默依赖一堆外部二进制 / 运行时**，缺任何一个对应功能就静默失败（opaque `ENOENT`），而用户根本不知道自己漏装了什么、怎么装。这是当前最大的「最后一公里」摩擦。

**实证（2026-06-04 runtime 调查）**：

- 一个从普通 shell 启动的 daemon，PATH 不含 `/opt/homebrew/bin` → `spawn ffprobe/ffmpeg ENOENT`，TTS 探时长、渲染、导出、波形全断；而用户机器其实**已经装了** ffmpeg@8.1.1。失败根因不是「没装」，是「app 依赖用户 shell 的 PATH，而那个 PATH 不可控」。
- TTS 走 `~/.autoviral/tts-venv` 的 edge-tts；ASR 走 stable-ts；趋势调研走 playwright chromium（日志 `Executable doesn't exist .../chromium_headless_shell`）。这些都要用户自己去琢磨安装。

**不做的代价**：用户装完 autoviral，create work、生成、导出里随机有一半功能静默坏掉，体验是「这软件怎么到处报错」，而不是「开箱即用」。

> 用户口径（2026-06-04）：「安装 autoviral 时，不管桌面端还是 npm/网页端，都应该同时帮用户装好所有依赖。现在网速很快，几个包不是问题，关键是减少用户麻烦，不能让用户自己去琢磨怎么装 ffmpeg。」

### 目标用户（for whom）

- **桌面端用户**（dmg / nsis）：双击安装，期望开箱即用，不开终端。
- **npm / CLI 用户**：`npm i -g autoviral` 后期望 `autoviral start` 直接能跑全部功能。

两类用户都**不应该**需要懂 ffmpeg / python / pip / playwright。

### 依赖清单（runtime 实测，非推测）

| 依赖 | 用途 | 候选安装机制 | 体量/难度 |
|---|---|---|---|
| **ffmpeg + ffprobe** | render / export / 波形(peaks) / TTS 探时长 | `ffmpeg-static` 或 `@ffmpeg-installer/ffmpeg`（npm 预编译，Mac/Win/Linux 全覆盖） | 易 · ~80MB · 装机即装 |
| **claude CLI**（Claude Code） | Chat 面板 agent 本体 `spawn("claude")` | 不能 bundle（Anthropic 的）→ 检测+引导，或迁 Claude Agent SDK 内嵌；**触及不变量 #4** | 中 · 架构决策（需 ADR） |
| **Python3 + edge-tts** | TTS 主路径（venv `~/.autoviral/tts-venv`） | bundle standalone Python（python-build-standalone）+ 首启建 venv，或依赖系统 Python | 中 · 几十 MB |
| **stable-ts** | ASR / 字幕 word-level | 同 venv，`pip install stable-ts`（注意不是 stable-whisper） | 中（+ whisper 模型 GB 级，首用下载） |
| **playwright chromium** | 趋势调研采集 | `playwright install chromium` 首启下载 | 重 · ~150MB |
| **yt-dlp** | youtube ingest | pip / 静态二进制 | 易 |

（**不同类**：OpenRouter API key 是 config secret 不是二进制 → 见 Out of scope。）

### In scope（本节明确承诺交付）

1. **共享 dependency-manager 模块**：detect → 下载到**受管位置** `~/.autoviral/bin/` + `~/.autoviral/tts-venv/`，app 一律从已知路径解析二进制，**不依赖用户 shell 的 PATH**。
2. **桌面端**：装机即就位核心二进制（ffmpeg/ffprobe via electron-builder `extraResources`）。
3. **npm 端**：`autoviral setup` / `autoviral doctor` 首启自检 + 带进度安装；**不用 postinstall**（常被禁、易碎）。
4. **分层策略**：核心、轻量依赖（ffmpeg / yt-dlp）装机即装；重型可选依赖（playwright ~150MB / whisper 模型 GB 级）**首次用到该功能时带进度懒装**。

### Out of scope / Non-Goals

- **API key（OpenRouter 等）配置** —— 那是 config secret，属另一条线（Settings / 不变量 #2），不在本节。
- **把 Python 依赖重写成 JS 实现** —— 除非顺带，不作为目标。
- **claude CLI 的多 backend / Chat 面板换 SDK** —— 本节只决定「claude 这个依赖怎么处理」的方向并开 ADR；多 backend Chat 本身按不变量 #4 仍 defer 到 0.2.0。

### 验收标准（acceptance criteria — 遵守不变量 #6：浏览器/产物可见为准）

> E2E 唯一通过标准是浏览器里看得到用户被许诺的结果，不是「装好了二进制」这种后端 artifact。详见 [`/.claude/rules/e2e-testing.md`](../../.claude/rules/e2e-testing.md)。

- [ ] **干净机器全链路**：在一台**未手动装过 ffmpeg / python / playwright** 的机器上装完 autoviral，**不做任何手动依赖安装**，能在浏览器里：create 视频 work → 生成女声 TTS 旁白（chat 气泡里能播放音频）→ 导出成片（导出进度走完、产物在 output/ 且 Studio 可见）。
- [ ] **不依赖系统 PATH**：从一个**精简 PATH**（不含 `/opt/homebrew/bin`）启动的 daemon，ffmpeg/ffprobe 仍可用（证明走的是受管位置解析，不是碰运气的 shell PATH）。
- [ ] **缺失即引导，不静默**：任一未就位依赖在首次触发时给出可见、可执行的提示 / 自动安装进度，而非裸 `ENOENT`。
- [ ] **重型懒装可见**：playwright / whisper 模型首用时有进度反馈，不是卡死的空白。

### 领域不变量（domain invariants — 不静默破坏）

- **#6 E2E=浏览器可见**：本需求的验收必须端到端走到浏览器可见产物，不能拿「二进制装好了」充数。
- **#1 Remotion 是渲染器**：渲染经 Remotion，底层用 ffmpeg —— 依赖自举必须保证 render-path 不被「找不到 ffmpeg」静默打断。
- **#4 Terminal agent skill-agnostic / Chat 是 claude-only**：claude CLI 作为依赖怎么处理，要与「Terminal 多 backend、Chat 暂 claude-only」的现状一致；任何 bundle/SDK 决策独立成 ADR。

### Issue 切片（待定稿后喂 `to-issues`，现仅为方向）

- 切片 A：dependency-manager 模块（detect + 受管位置解析，先覆盖 ffmpeg/ffprobe）—— tracer-bullet：精简 PATH 下渲染/导出仍通。
- 切片 B：桌面端 electron-builder `extraResources` bundle ffmpeg-static。
- 切片 C：`autoviral doctor` / `setup` 命令（首启自检 + 带进度安装）。
- 切片 D：Python venv 自举（edge-tts / stable-ts）+ 重型懒装策略。
- 切片 E：claude CLI 处理方向 ADR（检测引导 vs Agent SDK）。

### 风险 / Open Questions

- **claude CLI 无法 bundle**：是「检测 + 引导用户装」还是「迁到 Claude Agent SDK 内嵌掉这个依赖」？后者是较大架构变动，需 ADR。
- **核心 vs 懒装的分界线**：playwright（~150MB）/ whisper 模型（GB 级）强制装机会让安装体验很重 —— 分层边界要拍板。
- **Python bundle 策略**：standalone Python（python-build-standalone，体积大但零系统依赖）vs 依赖系统 Python（轻但用户机器可能没有 / 版本不对）。
- **跨平台二进制来源**：Win（nsis）/ Linux 的 ffmpeg / playwright / python 来源与签名。
- **离线 / 弱网**：用户「网速很快」的假设不总成立 —— 首启大下载失败时的回退与重试。

## §2 · TTS provider 网关一致性（Gemini-via-OpenRouter 主力 + edge fallback）

### Why / Problem Statement

TTS 是当前**唯一不走 OpenRouter 的能力**，违反不变量 #2「OpenRouter 是唯一外部网关 / 无直连 vendor」。`src/providers/tts/openai-tts.ts` 直连 `https://api.openai.com/v1/audio/speech`，是 ADR-007 OpenRouter 统一**之前**的遗留（registry 注释自承它是被改造成 provider 的 "OpenAI-only synth"）。这条 fallback 既不一致、又有潜在 bug：无 `OPENAI_API_KEY` 时回落 `OPENROUTER_API_KEY`，但 URL 仍打 `api.openai.com` → OpenRouter key 会被拒。

### 目标安排

| 角色 | provider | 网关 / 成本 |
|---|---|---|
| **主力** | `google/gemini-3.1-flash-tts-preview` | OpenRouter `/v1/audio/speech`，`OPENROUTER_API_KEY`，~$1/$20 per M token |
| **fallback** | edge-tts（**保留**） | 本地 Python CLI，免费、零 key（仍由 §1 自动安装，不消除） |
| **退役** | openai-direct | 删除 / 不再用 |

即把现有 fallback 链 `edge→openai` **翻转**为 **`Gemini(OpenRouter)→edge`**。

### 已验证的实现事实（2026-06-04 web 查证）

- `google/gemini-3.1-flash-tts-preview` 真实存在于 OpenRouter（2026-04-24 发布，70+ 语言，原生 PCM 24kHz/16-bit mono，SynthID 水印）。
- **OpenRouter 用 OpenAI 兼容的 `POST https://openrouter.ai/api/v1/audio/speech`**，`response_format` 支持 mp3/pcm/wav，输出 raw 音频流 → **不是重写**：现有 `openai-tts.ts` 几乎可直接改造（换 URL / key / model / `response_format:"mp3"`）。
- Gemini TTS 带 **200+ 内联情感标签**（`[whispers]`/`[laughs]`/`[excited]`），与现有 `edge-tts.ts` 的 `mapExpressiveTagsToSsml`（`[sigh]`/`[pause]`/`[whisper]`）同思路 → 可顺手统一表达层。

### In scope

1. 新建 `gemini-tts`（或 `openrouter-tts`）provider：`/v1/audio/speech` + `OPENROUTER_API_KEY` + model `google/gemini-3.1-flash-tts-preview` + `response_format:"mp3"`。
2. registry fallback 链翻转为 **Gemini 主力 → edge fallback**；退役 openai-direct。
3. voice 映射：请求侧 voice → Gemini 声音名（Zephyr / Puck / Kore… 实现时拉官方列表定表）。
4. **修两个 endpoint 的 fallback 不对称**：`POST /api/audio/tts`（agent 用，现 `pickProvider`→edge **only、无 fallback**）与 `POST /api/works/:id/tts`（dialog 用，有 `generateWithFallback`）对齐，都走 Gemini→edge。

### Out of scope / Non-Goals

- ElevenLabs / 火山 TTS（registry 里的 Phase 3.x 占位，不在本节）。
- 多 speaker / per-character Audio Profile（Gemini 支持，先不接）。

### 验收标准（不变量 #6：浏览器/产物可见为准）

- [ ] 配了 `OPENROUTER_API_KEY` 时，浏览器 dialog 配音 → 听到 **Gemini** 出的声（response `providerId` 为 gemini，chat / 素材库里音频可播放）。
- [ ] OpenRouter 失败 / 无 key 时 → 自动回落 **edge-tts** 仍出声（不是 500 空白）。
- [ ] agent 路径（`/api/audio/tts`）也享有同样的 Gemini→edge 兜底（不再 edge-only）。

### 领域不变量

- **#2 OpenRouter 唯一网关**：本节**修复** TTS 对 #2 的违反 —— 退役 `api.openai.com` 直连，TTS 改经 OpenRouter。修完同步更新 CONTEXT.md #2 措辞（当前文字已把 TTS 算作「走 OpenRouter」，属 aspirational，本节让它成真）。
- **#6 E2E=浏览器可见**：验收以听到声音 / 音频在 UI 可播放为准，不拿「endpoint 返回 200」充数。

### 风险 / Open Questions

- Gemini 原生 PCM；靠 `response_format:"mp3"` 规避转码，需确认 OpenRouter 该端点 mp3 稳定。
- 成本：Gemini ~$20/M output token vs edge 免费 —— 付费主力已经用户确认可接受（「免费的留作 fallback」）。
- voice 名映射表（Gemini 声音集 ≠ 现有 edge/OpenAI voice id）。
- openai-direct 的 key-fallback bug 随退役一并消失。

## §3 · 素材库 + 时间线交互（对齐 pro 视频编辑器）

### Why / Problem Statement

素材库与时间线的交互目前是「原语齐了，UX 层没接上」：核心 store 动作都在（`addClip` #78 / `moveClipToTrack` #88 kind-guarded / `removeAsset`），但用户要的「生成即见、能删、能拖」的直觉交互缺位 —— 现在只能靠 ＋ 按钮加素材、Inspector 下拉换轨、刷新页面才见新素材。目标对齐 CapCut / Premiere / Final Cut / DaVinci 的拖拽语义。

### 现状盘点（2026-06-04 grep 实证 —— 已有 vs 缺）

| 能力 | 现状 |
|---|---|
| 生成素材自动进库 | ❌ 缺（§3.1）—— `generate.ts` 不发事件 + `useBridgeEvents.ts` 无 `asset-added` case |
| 素材删除 | ⚠️ 半成品 —— store `removeAsset`（store.ts:718）在，但**无库内删除 UI**、**无 per-work 素材删除 endpoint**（只有 `DELETE /api/shared-assets/:cat/:file`）|
| 库→时间线 拖拽 | ❌ 缺 —— 时间线零 DnD（仅 Chat 附件有 `onDrop`）；加素材只有 #78 的 ＋ 按钮 |
| 同类型轨道间 拖拽移动 | ❌ 缺拖拽 —— #88 `moveClipToTrack`（kind-guarded）只接了 Inspector 下拉 |
| 类型约束原语 | ✅ 有 —— `moveClipToTrack` / `buildClipFromAsset` 已 kind-guard（#78/#88）|

### In scope

- **§3.1 生成素材自动进库**（bug，本会话 E2E 发现，详见 memory `project_asset_library_no_autorefresh`）：
  - (A) `generate.ts` image/video 成功后 `uiEventBus.publish({type:"asset-added", …})`（照 `audio.ts:279`）。
  - (B) `useBridgeEvents.ts` 加 `case "asset-added"` → 复用 `composition-changed` 的 refetch/invalidate。
- **§3.2 素材删除**：库内每个素材的删除控件 → 调 `removeAsset` + 新增 per-work asset 删除 endpoint（删盘上文件，对齐 `shared-assets` delete 的 SAFE_ID 守卫）。需处理「删除已被 clip 引用的素材」（级联 / 警告 / 禁止 —— 见 Open Q）。
- **§3.3 库→时间线拖拽**：从右侧素材库拖素材到中间下方轨道 → 调 `buildClipFromAsset` + `addClip`。**类型约束**：素材只能落对应类型轨道（video→video / audio→bgm / image→overlay），非法落点拒绝 + 视觉提示。
- **§3.4 同类型轨道间拖拽**：clip 在**同 kind** 的多条轨道间拖拽移动 → 调 `moveClipToTrack`（已 kind-guard）。跨 kind 禁止。
- **§3.5 交互对齐 pro 编辑器**：拖拽 ghost 预览 + drop 指示线 + 吸附（沿用 #59 carousel snap 思路）；非法落点视觉反馈；保留现有 ＋ 按钮 / Inspector 下拉作可达性备选，不删。

### Out of scope / Non-Goals

- 时间线 magnetic / ripple / insert-vs-overwrite 等高级编辑模式（先做基础拖拽）。
- 拖拽时跨 kind 的隐式素材转换（image→video 等）。

### 验收标准（不变量 #6：浏览器可见为准）

- [ ] 生成图/视频后，**不刷新**素材库即出现缩略图。
- [ ] 库内可删除素材，删后 UI 与磁盘都没了；被引用素材的删除有明确、可见的处理。
- [ ] 从库拖一个**视频**素材到**视频轨** → 出现 clip；拖到**音频轨**被拒（视觉提示）。
- [ ] 同为视频的两条轨道间拖一个 clip → 移动成功；拖到音频轨被拒。
- [ ] 拖拽过程有 ghost / drop 线 / 吸附，体感接近 CapCut / Premiere。

### 领域不变量

- **#3 deliverable yaml 是 SSoT**：所有拖拽 / 删除最终经 store → 原子写 `composition.yaml` + zod 校验，不旁路。
- **#6 E2E=浏览器可见**：拖完 / 删完以画面可见为准，不拿 store action 调用成功充数。

### 风险 / Open Questions

- 拖拽库选型：原生 HTML5 DnD vs dnd-kit / react-dnd（仓库现无 DnD 依赖，Chat 附件用原生）—— 一致性 + bundle 权衡。
- 类型↔轨道映射表要落定（image 落 overlay 还是新建 image 轨？参考 `buildClipFromAsset` 的 per-kind 规则）。
- 删除被引用素材：级联删 clip / 仅警告 / 禁止 —— 需拍板。
- 「对齐最好的视频编辑器」是开放标尺，定稿时须落成具体可验收的交互点（snap 容差、drop 指示、insert 行为），否则无法验收。

## §4 · agent 视觉自检 / 截屏工具（让 agent「看见」自己的产出）

### Why / Problem Statement

agent 现在是「盲做」：写 `composition.yaml` / `carousel.yaml`、调生成端点，但**看不到渲染结果**，只能假设 backend artifact 正确。这正是 `.claude/rules/e2e-testing.md` / 不变量 #6 反复警告的失败模式——「agent 写了文件 ≠ 用户看得到」。给 agent 一个截取当前画面的工具，让它：(1) 立刻看到用户当前的视角；(2) 交付前**自己视觉验证**产出有没有问题（slide 文字溢出 / clip 黑帧 / layer 错位…），把「最后一公里」闭环交给 agent 自己走。

### 现状盘点（2026-06-04 grep 实证）

| 能力 | 现状 |
|---|---|
| 视频单帧截图 | ❌ 缺 —— `autoviral render` 是 export 的 `--proxy` 别名，产**整段视频**非单帧；无 Remotion `renderStill` |
| carousel 页面 PNG | ⚠️ 部分 —— export 产 `output/*.png`（整套导出），无「当前 slide 即时截图」|
| 文本自检 | ✅ 有 —— `autoviral check`（composition lint findings），但**非视觉** |
| 浏览器画布截取 | ❌ 缺 —— 无 `toDataURL` over bridge |

### In scope

- 新 CLI 能力 `autoviral snapshot`（命名待定）：截取**当前画面**为 PNG，返回路径 → agent 用 Read 工具「看」（prompt 已教 agent Read 图片）。
  - **video**：Remotion `renderStill` 渲染当前 playhead（或 `--at <time>`）单帧 → PNG（比全片快几个量级）。
  - **carousel**：渲染当前 slide（或 `--slide <id>`）为 PNG（复用已有 carousel→PNG 路径）。
  - 截图落 `output/` 或临时目录，路径回 agent。
- agent prompt（`buildSystemPrompt`）补一句：交付前用 `autoviral snapshot` 视觉自检（呼应不变量 #6）。

### Out of scope / Non-Goals

- 录屏 / 多帧序列（先单帧）。
- 截浏览器 UI chrome（panel / 按钮）—— agent 要的是 canvas 内容不是 app 外壳。

### 验收标准（不变量 #6：浏览器可见为准）

- [ ] agent 在视频 work 跑 `autoviral snapshot` → 得当前帧 PNG，Read 后能「看到」画面内容（与 Studio 预览一致）。
- [ ] carousel work 同理得当前 slide PNG。
- [ ] agent 能据此**发现并修正**一个视觉问题（如文字溢出），修后 snapshot 复验通过。

### 领域不变量

- **#6 E2E=浏览器可见**：这个工具本身就是让 agent 能履行 #6 —— 视觉验证而非假设 artifact。
- **#4 Terminal agent skill-agnostic**：snapshot 走 `autoviral` CLI，任何 agent 通用，不绑 claude。

### 风险 / Open Questions

- **server-side `renderStill` vs 浏览器画布截取**：前者可靠（headless 也行）、是 deliverable 的渲染；后者更贴「用户 literal 视角」（含选中态 / 当前 slide）但需浏览器连着。倾向 `renderStill`（与现有 render-pipeline 同源），但「当前用户视角」的字面诉求要确认到哪个程度。
- `renderStill` 的字体 / asset 解析要与全片 render 一致（避免截图和成片不符）。
- 截图频率 / 成本（agent 频繁自检会不会拖慢；需不需要节流 / 缓存）。
- 命名：snapshot / screenshot / glance / preview-frame。

## §5 · 多对话 / 多终端会话（新建 + 保留 + 跳回）

> ⚠️ **触及 keystone 架构变更**：会话 keying 从 `workId` 改为 `(workId, sessionId)`，**超出 ADR-005 的 single-session 范围**。按 PRD 约定，需独立 **ADR-008**（或更新 ADR-005 Consequences 写明取代关系）锁定契约后再切 issue。

### Why / Problem Statement

用户要能「**新建一个对话 / 新建一个终端，同时保留原有的，方便跳回**」。今天是**单会话/工位**：一个 work 只有一个 chat（一个 cliSessionId）+ 一个 terminal（一个 pty），无法并存多个、无法跳回历史会话。

### 现状盘点（workflow 4-reader 实证，2026-06-04）

| 维度 | 现状 |
|---|---|
| Chat 会话 | `WsBridge.sessions: Map<workId, WsSession>` **仅按 workId keying**，一个 work 一个 in-memory session（单 cliProcess / messageHistory / cliSessionId），持久化 `chat.jsonl` + `work.yaml.cliSessionId`，可 `--resume`（`src/ws-bridge.ts:57-65,277`）|
| Terminal 会话 | 独立 bridge `/ws/terminal/{workId}` 起**一个 pty**（`pty_随机`，id 不暴露前端），**无 resume**，pty 退出要刷新页面（`src/server/terminal/terminal-ws.ts` / `pty-pool.ts`）|
| 前端切换 | RightPane 两按钮 tab（✦ Chat ｜ ⌨ Terminal），`useActiveSurface(workId)` + localStorage 切换，两面板都 mount（display:none 保状态），⌘\\ 切换（`RightPane/index.tsx:114-170`）|
| 文档 | **ADR-005 明确 scope 为「一个 work = 一个 Chat + 一个 Terminal」「no multi-tab concurrency」** —— 多会话不在其范围 |

**顺带发现的潜在 bug**：`/ws/terminal/{workId}` 是 1-to-1，**两个浏览器 tab 开同一个 work 会连到同一个 pty**（输出重复、resize 冲突）。多会话路由改造时应一并修。

### In scope（6 块，分层）

1. **会话 keying**：WS 路由 `/ws/browser/{workId}/{sessionId}`、`/ws/terminal/{workId}/{sessionId}`；服务端 `Map<workId, WsSession>` → 嵌套 `Map<workId, Map<sessionId, WsSession>>`，PtyPool 同理。
2. **持久化**：work.yaml 单 `cliSessionId` → 会话清单（**优先 `.sessions.jsonl` sidecar**，append-only，含 id / cliSessionId / createdAt / lastActive / preview；避免 inline 进 work.yaml 导致每次 `updateWork` 全量重写）。`chat.jsonl` → `chat-{sessionId}.jsonl`。**迁移**：旧单 cliSessionId → 第一条会话。
3. **前端会话列表 + 切换 UI**：Chat / Terminal 各自的会话 tab 条 / 下拉（「Session 2 · 3 小时前 · 12 turns」），「新建」按钮，点击切换（关旧 WS、开新 WS、按 sessionId 重 seed 历史）。
4. **前端状态重构**：`useChatSocket` / `useChatStore` / `useTerminalSocket` 接受 sessionId 参数（现假设单会话）。
5. **Terminal 生命周期**：新建终端不杀旧 pty；切走只 display:none，删除 tab 或 pty 退出才 dispose；顺带补 respawn 按钮。
6. **focus 通道**：现 `broadcastToBrowsers(workId)` 全广播 → 决定 focus（playhead / 选中）是跨会话共享还是按 sessionId 隔离（见 Open Q）。

### Out of scope / Non-Goals

- 跨 work 并发（多 work 同开）—— 本节只做单 work 内多会话。
- Terminal scrollback 跨重启持久化（先接受重开终端 = 新 shell）。
- 会话间内容搬运 / 合并。

### 验收标准（不变量 #6：浏览器可见）

- [ ] 在一个 work 里「新建对话」→ 出现第 2 个 chat tab，原对话**保留**，两者可点击跳回，各自历史正确。
- [ ] 「新建终端」→ 出现第 2 个终端，原终端 pty **不被杀**、scrollback 还在，可跳回。
- [ ] 刷新页面后会话清单恢复（sidecar 持久化生效）。
- [ ] 两个浏览器 tab 开同一 work 不再共用一个 pty（潜在 bug 修复）。

### 领域不变量

- **需 ADR-008**：会话 keying `(workId)`→`(workId,sessionId)` + 持久化模型 + focus 隔离，是**超出 ADR-005** 的 keystone 选择，按 PRD 约定独立成 ADR 再实现。
- **#4 Terminal agent skill-agnostic**：多终端仍走 pty + bridge 通用协议，不绑 claude。
- **#6 E2E=浏览器可见**：以「能新建、能跳回、历史正确」的画面为准。

### 风险 / Open Questions

- **会话爆炸**：无清理策略会攒上百个 stale 会话 → 需 TTL / archive / 删除 UI。内存：每活跃会话持 messageHistory ~60-90k。
- **work.yaml bloat** → 用 `.sessions.jsonl` sidecar（append-only），别 inline。
- **focus 共享 vs 隔离**：所有会话共用一个 playhead / 选中，还是每会话独立？UX 决策，ADR-008 拍。
- **resume 契约**：cliSessionId 是不可变 UUID，前后端 sessionId 方案要一致。
- **向后兼容迁移**：旧 work 单 cliSessionId → 默认首会话。
