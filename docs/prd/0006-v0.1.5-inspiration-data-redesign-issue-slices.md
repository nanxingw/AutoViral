### S1 · 接已有数据：渲染已在磁盘的 9 作品 per-work 表现 + 平均播放 KPI
**类型**：AFK · **覆盖 US**：1,2 · **deep 模块**：D1

**What to build**：数据页当前坐在一份冻结的真实抖音抓取上，9 件作品的 per-post 指标（播放/点赞/评论/分享/收藏）+ `avg_play` 早已在磁盘、且已被适配器解析成 `works[]`/`avgPlay`，却从没上屏——这是 built-not-wired。建 D1 派生纯函数核心（输入 `works[]`+`summary` → 平均播放、可排序的每件作品表现行），并把它渲染成一张可按播放/点赞排序的作品表现表 + 一个「平均播放」KPI。冻结的 `latest.json` 即天然测试 fixture。

**Acceptance criteria**：
- [ ] D1 派生核心有 TDD 测试：以 `latest.json` 为 fixture，断言平均播放=624、最高播放作品正确、排序正确。
- [ ] 数据页渲染 9 件作品真实指标，可按播放/点赞排序（Workflow 浏览器 E2E：截图 + DOM textContent 二确数字非占位）。
- [ ] 平均播放 KPI 与现有 平均点赞/评论/互动率 并列显示。

**Blocked by**：None — can start immediately

### S2 · 删假承诺：删人口卡 + 诚实三段式空态 + 平台诚实矩阵
**类型**：AFK · **覆盖 US**：8,9,10,11

**What to build**：年龄/性别/地域三张卡 + 最新洞察空卡读的字段无任何代码写过、且是 OAuth-only 私有数据（用户平台 5 粉规模下任何 API 都不返回）——「等待后台采集首批样本」是架构性谎言。**删掉**这些卡（不是推迟），换成可复用的诚实三段式空态组件：Inform（为什么空）/ Inspire（带清晰水印的示例形状，绝不与真实数据混淆）/ Activate（一个真实可行 CTA，区分「连接账号」vs「粉丝还不够」vs「该平台无 API」）。新增一张「平台诚实矩阵」列出每平台 {能否自有数据 / 能否人口属性·门槛 / 趋势是实采还是 LLM}。删掉指向 501 的「刷新」死胡同 CTA。

**Acceptance criteria**：
- [ ] 数据页不再出现任何"等待后台采集"的永空卡（Workflow E2E：DOM 查询确认这些卡已移除）。
- [ ] 空态三段齐全；示例形状带可见水印（computed-style 二确水印样式，避免被误认作真实数据）。
- [ ] 平台诚实矩阵渲染 4 平台真实能力（E2E textContent 二确文案）。
- [ ] 不再有 CTA 指向已 501 的 refresh。

**Blocked by**：None — can start immediately

### S3 · D2 benchmark 百分位带：让孤立 KPI 变诊断句
**类型**：AFK · **覆盖 US**：3,4 · **deep 模块**：D2

**What to build**：把 KPI 放进同粉丝层创作者的基线带里，让"互动率 2.6%"变成"低于 nano 层中位数，目标区间 X–Y"。建 D2 模块：仓库内静态基线 JSON（per 平台 × 粉丝层）+ 定位纯函数 `(platform, 粉丝层, metric, value) → {带位置, 诊断文案, 安抚文案}`。**关键约束**：必须为用户真实平台（抖音）配 platform-correct 带，否则就近用别平台基线是「伪诊断」、反破坏诚实；若无可信抖音公开基线，明确标注「参考性、非你所在平台」。附"小账号天然互动更高"的安抚文案。

**Acceptance criteria**：
- [ ] D2 有 TDD 测试：断言带位置/诊断文案正确；**专门断言抖音的带是 platform-correct 或被明确标注为参考性**（防伪诊断回归）。
- [ ] 每个 KPI 旁渲染基线带 + 诊断 + 安抚（Workflow E2E：DOM 二确文案）。

**Blocked by**：None（与 S1 的 KPI 组合，但 D2 可独立先行）

### S5 · 采集器接线：collector 服务 + un-501 refresh + cookie 授权 UI
**类型**：**HITL**（首跑需用户在浏览器登录抖音才能完成真 scrape E2E）· **覆盖 US**：12,14 · **deep 模块**：D4（接线层）

**What to build**：恢复 f2 + browser_cookie3 采集器，接口 `refresh(profileUrl) → CreatorData | CollectorError`（解析边界 raw f2 JSON → CreatorData 纯函数、可 fixture 测；实际抓取走集成）。把 `POST /api/analytics/refresh` 从硬 501 改为真刷新。Settings 加抖音 profile URL 配置 + cookie 授权流（读浏览器已登录 sessionid）+ 刷新按钮。失败（cookie 过期/改版）→ 结构化 `CollectorError` + 清楚可操作提示（提示重新登录）。明确 token 仅本地、失败如何诚实降级。

**Acceptance criteria**：
- [ ] D4 解析边界有 TDD 测试：raw f2 JSON fixture → 正确 CreatorData；过期 cookie → 结构化 CollectorError。
- [ ] refresh 不再返 501；真刷新后 `latest.json` 更新、数据页（S1 的表）反映新数（Workflow E2E + 用户提供登录态）。
- [ ] cookie 过期时 UI 显示可操作的"重新登录"提示，非静默空页。

**Blocked by**：S4

### S6 · D5 coach 后端：可持久「研究/策略」session kind + persona + session 级 model
**类型**：AFK · **覆盖 US**：18,22,23,24 · **deep 模块**：D5

- [ ] 后端可被驱动并流式返回（可用现有 refresh-stream 式路径或新端点验证）；对薄数据诚实（数据太少则说太少、转靠趋势/兴趣）。
- [ ] 成本护栏：单测断言上下文惰性加载、token 预算生效。
