# Phase 8.3 — Speed Ramp / Time Remap Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Today every `VideoClip` plays at 1× — its source MP4 is sliced via `startFrom`/`endAt` and rendered at the project fps, period. Phase 8.3 turns clip playback **speed** into a *keyframable* property: a clip can hold steady at 0.5× for the first second, ramp linearly up to 2× over the next two seconds, and stop on a freeze-frame at 0.1×, all without re-encoding the source asset. The Inspector's existing `KeyframePanel` (shipped in 8.2) gains `"speed"` as a property option for VideoClip; the Remotion renderer reads `interpolateProperty(c.keyframes, "speed", localSec) ?? 1.0` every frame and feeds it to `<OffthreadVideo playbackRate={speed} />`; export-side ffmpeg post-process applies `setpts=PTS/<speed>` (static-speed v1) for byte-accurate exports.

The acceptance video is a `VideoClip` of original duration 4s with two `speed` keyframes — `t=0 v=2.0, t=4 v=2.0` — rendering as a clip whose **timeline duration is 2s** (twice as fast, so half the elapsed time) in *both* the live preview and the exported MP4. A second AC inverts: `speed=0.5` → 8s timeline duration.

**Architecture:** Phase 8.3 reuses the entire 8.2 keyframe stack — schema, `interpolateProperty`, store actions, KeyframePanel rows. The only schema change is *adding a single string `"speed"`* to `KeyframePropertySchema` enum (D2). The renderer change is *one new pure helper* `computeVideoSpeedForFrame(clip, localFrame, fps): number` that wraps `interpolateProperty(c.keyframes, "speed", localSec) ?? 1.0` with a `[0.1, 4.0]` clamp (D4). The renderer hands that number to `<OffthreadVideo playbackRate={speed}>`. Timeline duration is computed by a new helper `effectiveClipDuration(clip): number` that integrates the speed curve numerically — for static `speed=k` it's `(out - in) / k`; for variable speed it samples and integrates. Server-side, render-pipeline gains a *pre-Remotion* stage that, for any clip whose **single static speed** ≠ 1.0, invokes ffmpeg with `setpts=PTS/<speed>` + `atempo=<speed>` and rewrites the comp's clip src to the resampled file (v1 limitation: variable-speed export is deferred to v2 alongside ffmpeg `setpts='PTS*expr'` work).

**Tech Stack:**
- Schema: existing `zod` enum in `src/shared/composition.ts` — one-line addition to `KeyframePropertySchema`.
- Pure helpers: `src/shared/keyframes.ts` already has `interpolateProperty`. New helpers (`computeVideoSpeedForFrame`, `effectiveClipDuration`) live in a small new file `src/shared/speed-ramp.ts` so server (render pipeline) and web (timeline width, renderer) both consume them.
- Remotion: `<OffthreadVideo playbackRate={number} />` — already validated in `node_modules/remotion/dist/cjs/validate-media-props.js` (confirmed during baseline audit). No additional Remotion API needed.
- ffmpeg: `setpts=PTS/<speed>` (video) + `atempo=<speed>` (audio) — for v1 *static* speed only. The `atempo` filter caps individual values at 0.5..2.0; for our `[0.1, 4.0]` range we chain them (`atempo=2.0,atempo=2.0` for 4×, `atempo=0.5,atempo=0.5` for 0.25× etc.). A small helper `chainAtempo(speed)` builds the chain.
- Store: existing `addKeyframe` / `removeKeyframe` / `updateKeyframe` actions handle `"speed"` keyframes with **zero new code** (D2). Only one regression test verifies this.
- UI: `KeyframePanel`'s `propertiesForClip("video")` adds `"speed"` to the dropdown. ~2 LOC change.
- Tests: Vitest + Testing Library, same conventions as Phase 8.2.

| **D9** | **Direction of integration: speed > 1 SHRINKS timeline duration; speed < 1 EXPANDS it.** Math sanity: if a 4s source plays at speed=2, it finishes in 2s of timeline → output duration is 2s. To consume the full `(out - in)` seconds of source given a time-varying `speed(t_timeline)`, we solve `∫₀^T speed(τ) dτ = (out - in)` for T. The naive approach `effectiveClipDuration = ∫ 1/speed(t_source) dt_source` is *also* equivalent if we parameterize `speed` over source time, but the keyframe times in this plan are **timeline-time**, not source-time. **D9 pins this:** Phase 8.3 keyframe `time` values are in **timeline-relative seconds from clip start** (i.e. clip-local *output* time), so `effectiveClipDuration` is the T satisfying the integral above. We solve via incremental sampling: walk timeline-time at 0.01 s steps, accumulating consumed source time; stop when consumed source time reaches `(out - in)`; T at that step is `effectiveClipDuration`. This is the same convention 8.2 uses for `scale` — keyframe times are clip-local timeline seconds. ## Task 8.3.A — Schema extension + speed-ramp pure helpers

**Goal:** Land the contract. After this task: (1) `Composition` validates with `"speed"` keyframes — values constrained to `[0.1, 4.0]` server-side; (2) `effectiveClipDuration(clip)` returns the timeline-width seconds for any clip; (3) `computeVideoSpeedForFrame(clip, frame, fps)` returns the per-frame playbackRate; (4) `isStaticSpeed(clip)` returns the constant `k` if all speed keyframes agree, else `null`.

**Pitfalls:**
- Adding `"speed"` to `KeyframePropertySchema` is a single-string change but it also *implicitly extends* every clip schema's allowed keyframes. We must add D10's `superRefine` on each clip schema (Video/Audio/Overlay) so that an out-of-range speed keyframe is rejected at parse time. Resist the urge to put the `[0.1, 4.0]` constraint inside `KeyframeSchema.refine` — that would reject a perfectly valid `scale=10` keyframe (scale > 4) as if it were a speed value.
- `effectiveClipDuration` for a static-speed clip = `(out - in) / speed`. For variable, the brief specifies "integrate the speed curve". We use the form D9 pins down: `T` such that `∫₀^T speed(τ) dτ = (out - in)`, where `speed(τ)` is `interpolateProperty(kfs, "speed", τ)`. Closed-form for piecewise-linear segments exists (trapezoid integration), but for v1 we use **dense sampling at 0.01 s** and stop when accumulated source-time hits `(out - in)`. This is O(n) where n = `effectiveDur * 100` — for a 60s clip at 0.5× that's ~12k samples, takes ~100µs in JS. Do **not** use a closed-form per-segment integral in v1 — adds 60 LOC for marginal speed gain.
- `clampSpeed(v)` clamps to `[0.1, 4.0]` and is the *runtime* guard. The schema `superRefine` is the *parse-time* guard. Both exist. Tests cover both paths.
- `isStaticSpeed(clip)` returns `null` when *no* speed keyframes exist (so the caller knows to use 1.0 fallback) AND when keyframes disagree. When keyframes all agree (within `1e-4`), it returns the agreed value. Renderer uses this to pick "static-fast-path" vs "per-frame-recompute".
- Existing 8.2 `interpolateProperty` already handles the `"speed"` enum entry without modification — it's a generic `KeyframeProperty` consumer. **Do not** modify `interpolateProperty`. Just verify a regression test.

### Step 3: Implement `speed-ramp.ts`

- [ ] Create `src/shared/speed-ramp.ts`:

  ```ts
  import type { Keyframe } from "./composition.js";
  import { interpolateProperty, KEYFRAME_TIME_EPSILON } from "./keyframes.js";
  import { SPEED_MIN, SPEED_MAX } from "./composition.js";

  /** Defensive runtime guard. D4. */
  export function clampSpeed(v: number): number {
    if (!Number.isFinite(v)) return 1.0;
    return Math.max(SPEED_MIN, Math.min(SPEED_MAX, v));
  }

  /**
   * Per-frame playback rate for a VideoClip. Reads the "speed" keyframes (if any)
   * via interpolateProperty and falls back to 1.0 (D3). Output is clamped to
   * [SPEED_MIN, SPEED_MAX] (D4).
   */
  export function computeVideoSpeedForFrame(
    clip: { keyframes?: readonly Keyframe[] },
    localFrame: number,
    fps: number,
  ): number {
    const localSec = localFrame / fps;
    const v = interpolateProperty(clip.keyframes, "speed", localSec);
    return clampSpeed(v ?? 1.0);
  }

  /**
   * Returns the agreed-upon speed value if every speed keyframe on the clip has
   * the same value (within KEYFRAME_TIME_EPSILON), otherwise null. Null also
   * means "no speed keyframes" — the caller treats null as "static 1.0".
   *
   * Used by the export pipeline (D6) to decide between fast-path ffmpeg setpts
   * and the deferred variable-speed path.
   */
  export function isStaticSpeed(clip: { keyframes?: readonly Keyframe[] }): number | null {
    const speedKfs = (clip.keyframes ?? []).filter((k) => k.property === "speed");
    if (speedKfs.length === 0) return null;
    const first = speedKfs[0].value;
    for (const k of speedKfs) {
      if (Math.abs(k.value - first) > KEYFRAME_TIME_EPSILON) return null;
    }
    return clampSpeed(first);
  }

  /**
   * Timeline width (in seconds) of a clip, accounting for speed ramps (D7).
   *
   * Convention (D9): keyframe `time` values are clip-local *timeline* seconds.
   * For static speed=k, width = (out - in) / k. For variable speed, we sample
   * the speed curve at 0.01s steps along timeline-time, accumulating consumed
   * source-time; the timeline-time at which we have consumed (out - in) seconds
   * of source IS the effective clip duration.
   */
  export function effectiveClipDuration(clip: {
    in: number;
    out: number;
    keyframes?: readonly Keyframe[];
  }): number {
    const sourceDur = clip.out - clip.in;
    if (sourceDur <= 0) return 0;

    const stat = isStaticSpeed(clip);
    if (stat !== null) return sourceDur / stat;

    const hasSpeedKfs = (clip.keyframes ?? []).some((k) => k.property === "speed");
    if (!hasSpeedKfs) return sourceDur;

    const dt = 0.01; // 100 Hz sampling — fine for any real curve
    let timelineT = 0;
    let consumed = 0;
    // Cap to a generous upper bound to prevent runaway loops on pathological inputs.
    const MAX_TIMELINE_T = sourceDur / SPEED_MIN + 1; // sourceDur at slowest = max timeline
    while (consumed < sourceDur && timelineT < MAX_TIMELINE_T) {
      const speed = clampSpeed(
        interpolateProperty(clip.keyframes, "speed", timelineT) ?? 1.0,
      );
      const stepConsumed = speed * dt;
      if (consumed + stepConsumed >= sourceDur) {
        // Linear interpolation within the final step for sub-step accuracy.
        const remaining = sourceDur - consumed;
        timelineT += remaining / speed;
        return timelineT;
      }
      consumed += stepConsumed;
      timelineT += dt;
    }
    return timelineT;
  }
  ```

### Step 4: Renderer code reference (full updated VideoClipRenderer)

```tsx
import { Sequence, OffthreadVideo, useVideoConfig, useCurrentFrame } from "remotion";
import type { VideoClip, Track } from "../../types";
import { toCssFilter } from "../filters/cssFilters";
import { interpolateProperty } from "@shared/keyframes";
import { computeVideoSpeedForFrame, effectiveClipDuration } from "@shared/speed-ramp";

export function computeVideoTransformForFrame(
  clip: VideoClip,
  localFrame: number,
  fps: number,
): { scale: number; x: number; y: number; rotation: number } {
  const localSec = localFrame / fps;
  const t = clip.transforms;
  const kfs = clip.keyframes;
  // "speed" keyframes feed playbackRate (see VideoClipRenderer); they are NOT a CSS scale (D8).
  return {
    scale: interpolateProperty(kfs, "scale", localSec) ?? t.scale,
    x: interpolateProperty(kfs, "x", localSec) ?? t.x,
    y: interpolateProperty(kfs, "y", localSec) ?? t.y,
    rotation: interpolateProperty(kfs, "rotation", localSec) ?? t.rotation,
  };
}

function VideoClipRenderer({ clip }: { clip: VideoClip }) {
  const { fps } = useVideoConfig();
  const frame = useCurrentFrame();
  const filter = toCssFilter(clip.filters);
  const { scale, x, y, rotation } = computeVideoTransformForFrame(clip, frame, fps);
  const speed = computeVideoSpeedForFrame(clip, frame, fps);
  return (
    <OffthreadVideo
      src={clip.src}
      startFrom={Math.round(clip.in * fps)}
      endAt={Math.round(clip.out * fps)}
      playbackRate={speed}
      style={{
        width: "100%",
        height: "100%",
        objectFit: "cover",
        filter: filter || undefined,
        transform: `translate(${x}px, ${y}px) rotate(${rotation}deg) scale(${scale})`,
      }}
    />
  );
}

export function VideoTrackRenderer({ track }: { track: Track }) {
  const { fps } = useVideoConfig();
  if (track.hidden) return null;
  return (
    <>
      {(track.clips as VideoClip[]).map((c) => {
        const from = Math.round(c.trackOffset * fps);
        const dur = Math.max(1, Math.round(effectiveClipDuration(c) * fps));
        return (
          <Sequence key={c.id} from={from} durationInFrames={dur}>
            <VideoClipRenderer clip={c} />
          </Sequence>
        );
      })}
    </>
  );
}
```

## Task 8.3.E — Server-side ffmpeg setpts integration in render-pipeline

**Goal:** For static-speed clips, the export pipeline pre-processes each affected source MP4 with `setpts=PTS/<speed>` + chained `atempo` *before* Remotion renders the composition. The deep-cloned comp's `clip.src` is rewritten to the resampled file. For variable-speed clips, the pipeline emits a console warning and renders at 1× (D6). Tests mock `child_process.spawn` per existing render-pipeline.test.ts conventions.

**Pitfalls:**
- ffmpeg `atempo` filter individual instances are constrained to `[0.5, 2.0]`. To reach 4× we chain `atempo=2.0,atempo=2.0`; for 0.25× we chain `atempo=0.5,atempo=0.5`. Our range is `[0.1, 4.0]` so worst case is 0.1× = `atempo=0.5,atempo=0.5,atempo=0.4` (need 3-step chain — 0.5*0.5*0.4 = 0.1). Helper `chainAtempo(speed: number): string` returns the comma-joined filter string.
- **Pre-Remotion vs post-Remotion:** D6 says *pre-Remotion*. Why? Because Remotion's `<OffthreadVideo playbackRate>` *will* play the source at the right rate in *preview*, and Remotion *can* render at that rate too — but the rendered output's framerate matches `composition.fps`, which means Remotion would internally drop / duplicate frames. For a clean 2× export from a 30fps source to a 30fps timeline, the clean way is: produce a new asset where the 30fps source is resampled at 60fps source-time → 30fps timeline-time. ffmpeg `setpts=PTS/2` does exactly this. Remotion then renders the resampled file at 1× and the math works out. Doing it post-Remotion would mean re-encoding the entire composition's video stream — 10× more expensive on a 60s comp.
- **Caching the resampled file**: `clip-{id}-speed-{round(speed*100)}.mp4` keyed by clip id + speed. If exists, skip the ffmpeg pass. This is critical — re-running the pipeline shouldn't re-encode every clip every time.
- **Comp deep-clone**: don't mutate the caller's comp. Build a new comp with rewritten srcs.
- **Skipping sourceless clips**: if a clip has speed=1 (no keyframes or static==1), skip the pre-pass entirely and use the original src.

### Step 3: Implement `speed-ramp-ffmpeg.ts` + pipeline integration

- [ ] Create `src/server/speed-ramp-ffmpeg.ts`:

  ```ts
  import { spawn } from "node:child_process";
  import { stat } from "node:fs/promises";
  import { join, basename, dirname } from "node:path";
  import type { Composition, VideoClip } from "../shared/composition.js";
  import { isStaticSpeed, effectiveClipDuration } from "../shared/speed-ramp.js";

  /**
   * Build the comma-chained atempo filter expression for any speed in [0.1, 4.0].
   * atempo's per-instance range is [0.5, 2.0], so chains decompose:
   *   - speed >= 1: repeated 2.0 + final adjustment in [1.0, 2.0]
   *   - speed <  1: repeated 0.5 + final adjustment in [0.5, 1.0]
   * For speed exactly 1.0 returns the no-op "atempo=1.0".
   */
  export function chainAtempo(speed: number): string {
    if (speed === 1.0) return "atempo=1.0";
    const parts: number[] = [];
    let remaining = speed;
    if (speed > 1.0) {
      while (remaining > 2.0 + 1e-9) {
        parts.push(2.0);
        remaining /= 2.0;
      }
    } else {
      while (remaining < 0.5 - 1e-9) {
        parts.push(0.5);
        remaining /= 0.5;
      }
    }
    parts.push(remaining);
    return parts.map((p) => `atempo=${p.toFixed(4)}`).join(",");
  }

  /**
   * Build the ffmpeg argv for a single clip's speed pre-pass:
   *   ffmpeg -y -i {input} -filter_complex "[0:v]setpts=PTS/{speed}[v];[0:a]<chainAtempo>[a]" -map "[v]" -map "[a]" {output}
   */
  export function buildSpeedRampFilterArgs(
    input: string,
    output: string,
    speed: number,
  ): string[] {
    const atempo = chainAtempo(speed);
    return [
      "-y", "-loglevel", "error",
      "-i", input,
      "-filter_complex",
      `[0:v]setpts=PTS/${speed}[v];[0:a]${atempo}[a]`,
      "-map", "[v]", "-map", "[a]",
      output,
    ];
  }

  /** Spawn ffmpeg + collect stderr + reject on non-zero. AbortSignal-aware. */
  export async function runSpeedRampPass(
    input: string,
    output: string,
    speed: number,
    signal?: AbortSignal,
  ): Promise<void> {
    const args = buildSpeedRampFilterArgs(input, output, speed);
    return new Promise<void>((resolve, reject) => {
      if (signal?.aborted) {
        reject(new Error("runSpeedRampPass: aborted before spawn"));
        return;
      }
      const child = spawn("ffmpeg", args);
      let stderr = "";
      child.stderr.on("data", (b: Buffer | string) => { stderr += b.toString(); });
      const onAbort = () => { try { child.kill("SIGTERM"); } catch { /* ignore */ } };
      signal?.addEventListener("abort", onAbort, { once: true });
      child.on("close", (code: number | null) => {
        signal?.removeEventListener("abort", onAbort);
        if (signal?.aborted) reject(new Error("runSpeedRampPass: aborted"));
        else if (code === 0) resolve();
        else reject(new Error(`runSpeedRampPass: ffmpeg exit ${code}\n${stderr}`));
      });
      child.on("error", (err) => {
        signal?.removeEventListener("abort", onAbort);
        reject(err);
      });
    });
  }

  /**
   * Pre-Remotion stage. For each VideoClip with a STATIC non-1 speed, runs the
   * setpts/atempo pass and rewrites clip.src to the resampled file. For
   * VARIABLE speed (D6), logs a one-time warning and leaves clip.src unchanged.
   *
   * Returns a new composition; never mutates the input.
   */
  export async function applySpeedRampPrePass(
    comp: Composition,
    workDir: string,
    signal?: AbortSignal,
  ): Promise<Composition> {
    const newTracks = await Promise.all(
      comp.tracks.map(async (track) => {
        if (track.kind !== "video") return track;
        const newClips = await Promise.all(
          (track.clips as VideoClip[]).map(async (c) => {
            const stat_speed = isStaticSpeed(c);
            const hasVariable = (c.keyframes ?? []).some((k) => k.property === "speed") && stat_speed === null;
            if (hasVariable) {
              console.warn(
                `[render-pipeline] Variable-speed export not supported in v1 (clip ${c.id}); ` +
                "rendering at 1×. Variable-speed export ships in 8.3.5.",
              );
              return c;
            }
            if (stat_speed === null || Math.abs(stat_speed - 1.0) < 1e-4) {
              return c; // no speed or speed=1 → no-op
            }
            const inputBase = basename(c.src);
            const cacheName = `clip-${c.id}-speed-${Math.round(stat_speed * 100)}.mp4`;
            const cachePath = join(workDir, cacheName);
            // Cache hit?
            try {
              await stat(cachePath);
              return { ...c, src: cachePath };
            } catch {
              // miss — run the pass
            }
            await runSpeedRampPass(c.src, cachePath, stat_speed, signal);
            return { ...c, src: cachePath };
          }),
        );
        return { ...track, clips: newClips };
      }),
    );
    return { ...comp, tracks: newTracks };
  }
  ```
