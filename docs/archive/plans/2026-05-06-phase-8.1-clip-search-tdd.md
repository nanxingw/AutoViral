# Phase 8.1 — CLIP Semantic Search Over Local Asset Library — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Today the AssetSidebar's `LibraryTab` lists every asset under a work's `assets/` and `output/` directories, bucketed by file type (CLIPS / IMAGES / AUDIO / TEXT). Filtering is filename-substring only — searching "panda" only finds files literally named `*panda*`. Phase 8.1 introduces **semantic** asset search backed by OpenCLIP image/text embeddings: the user types "panda" into a new SearchBox, the server returns the top-K visually-most-panda assets ranked by cosine similarity, regardless of how the file is named. The whole stack ships with a **stub mode** so a teammate without Python ML deps installed still gets a usable build (search returns `{ results: [], stub: true }` and the UI shows a friendly "Semantic search unavailable — install dependencies" hint).

The acceptance video is: a work containing 8 images (one of which is `asset-bamboo-eater-1.png`, visibly a panda) plus a few unrelated clips. The user clicks **Build index** in the SearchBox panel, waits ~30s for embeddings to compute, types `panda`, and sees `asset-bamboo-eater-1.png` ranked #1 with a similarity score around `0.30+` — even though the filename never says "panda". A second AC verifies stub mode: with `open_clip` import disabled (env flag), the SearchBox renders a "build dependencies" banner and the search call returns `{ results: [], stub: true, reason: "open_clip_torch not installed" }` with HTTP 200.

**Architecture:** Phase 8.1 is intentionally split into a *skinny* Python ML core (encode-and-rank, ~300 LOC across `build_index.py` + `search.py`), a Node↔Python bridge that re-uses the existing `runPythonScript` (Phase 6 smart-crop pattern), and one new React component (`SearchBox.tsx`) mounted inside the existing `LibraryTab`. Per-work index storage lives at `~/.autoviral/works/<workId>/clip-index/{embeddings.npy, asset-uris.json, meta.json}` (D3) so different works can carry different libraries without crossing wires. The default model is **ViT-B/32** (~150MB, D1) because the master plan's ViT-L/14 (~1.7GB) is too heavy for the implementer's machine and the quality delta is small for short-video gallery search; ViT-L/14 is opt-in via `AUTOVIRAL_CLIP_MODEL=ViT-L-14`. Stub mode (D4) trips automatically on `ImportError: open_clip_torch` — the server endpoints stay HTTP 200 and emit `{ stub: true }` so the UI degrades gracefully rather than 500-ing.

**Tech Stack:**
- Python ML: `open_clip_torch` (HuggingFace OpenCLIP fork; pure-Python, no CUDA required, CPU-only inference is acceptable for the < 100-asset libraries this phase targets), `torch>=2.0` (already installed at v2.9.0 on dev machine), `Pillow>=10` (already installed), `faiss-cpu>=1.13` (already installed at v1.13.2, confirmed during baseline audit), `numpy` (transitive). Pinned in `requirements.txt`.
- Node bridge: existing `src/server/python-bridge.ts` `runPythonScript<T>()` — JSON-on-last-stdout-line contract (Phase 6 proven). No new dep.
- ffmpeg: existing `/opt/homebrew/bin/ffmpeg` v8.1 (audited). Used for video mid-frame extraction (D6) via `ffmpeg -ss <t> -i <video> -frames:v 1 -y <jpg>`.
- Server: `Hono` (existing). Two new routes: `POST /api/clip-index/build` and `GET /api/works/:id/assets/search`.
- Web: existing `@tanstack/react-query` for fetching, no new dep. `useDebouncedValue` hook (write a tiny one if absent — 8 LOC).
- Tests: Vitest + Testing Library on web side (same as Phase 5/6/7/8.2/8.3); pytest for the Python scripts; Vitest with mocked `runPythonScript` on server side. No real CLIP weights touched in CI.

## Task 8.1.A — Python ML core: build + search + frame extract

**Goal:** Land the offline ML stack. After this task, a developer can run `python build_index.py --asset-list /tmp/assets.json --workid <id> --output-dir ~/.autoviral/works/<id>/clip-index/` and get a working `embeddings.npy` + `asset-uris.json` + `meta.json` triple; then `python search.py --query "panda" --workid <id> --topk 5` returns ranked URIs as JSON. ~17 pytest tests cover the full surface including stub mode.

**Pitfalls:**
- `open_clip.create_model_and_transforms("ViT-B-32", pretrained="laion2b_s34b_b79k")` is the canonical loader call. The `pretrained=` slug must match a HuggingFace OpenCLIP weights tag exactly — typos download nothing and silently fall back to random weights. We pin both model name AND pretrain slug in `MODEL_REGISTRY`.
- `torch.no_grad()` is non-optional around encode calls — without it, gradient buffers will balloon memory by ~2× per inference.
- CLIP image encoder requires images normalized to 224×224 (B/32) or 336×336 (L/14, but B/32 is the default). The `transforms` returned by `create_model_and_transforms()` does this — DON'T hand-roll the resize. Pass through `transforms(Image.open(p).convert("RGB"))`.
- Cosine similarity in faiss: build with `IndexFlatIP` (inner product) and **L2-normalize the vectors before indexing** (`embeddings = embeddings / np.linalg.norm(embeddings, axis=1, keepdims=True)`). After normalization, inner product == cosine similarity. Forgetting to normalize → wrong rankings.
- ffmpeg `-ss <t>` placement matters for accuracy: put `-ss` BEFORE `-i` for fast (keyframe-snapped) seek, AFTER for accurate-but-slow seek. For mid-frame extraction we use the fast form (D6 just needs "a representative frame", not a specific timestamp). Add `-y` to overwrite without prompting (avoids stdin-blocking in `runPythonScript`).
- ffprobe output for `format=duration` is a string with a trailing newline; `float(line.strip())` is the parse. If `format=duration` returns `N/A` (some webm variants), fall back to `stream=duration` on the first video stream. If both fail → skip the asset with a stderr warning per D6.
- `--asset-list` JSON parse errors must abort the whole build (exit 0, JSON body `{ stub: true, reason: "invalid_asset_list", error: <msg> }` per D4/D11) — NOT exit non-zero. Otherwise the server has to special-case malformed input as "operational error" vs. "real crash".
- Atomic rename of `embeddings.npy` (D14): write to `embeddings.npy.tmp`, then `os.replace(tmp, final)`. `os.replace` is atomic on POSIX; `os.rename` is *not* if the destination exists on Windows but we don't target Windows.
- The pytest fixture for stubbed open_clip must monkeypatch `open_clip.create_model_and_transforms` AND `open_clip.tokenize` (search.py uses both). The stub model's `encode_image(t)` should return a deterministic `torch.tensor` of shape `(B, 512)` derived from a hash of the input — e.g. `torch.tensor([[float(hash(str(b.shape))) % 1000 / 1000.0] * 512 for b in batch])` — so tests can assert "image A and image B get different vectors" without caring about absolute values.

  ```ts
  import { useEffect, useState } from "react";
  export function useDebouncedValue<T>(value: T, delayMs: number): T {
    const [debounced, setDebounced] = useState(value);
    useEffect(() => {
      const t = setTimeout(() => setDebounced(value), delayMs);
      return () => clearTimeout(t);
    }, [value, delayMs]);
    return debounced;
  }
  ```

  ```ts
  export interface ClipSearchResponse { results: ClipSearchResult[]; stub: boolean; reason?: string; searchMs: number; }
  export interface ClipIndexStatus { stub: boolean; reason?: string; model?: string; assetCount?: number; indexedAt?: string; }

  export function useClipIndexStatus(workId: string | null) {
    return useQuery({
      queryKey: ["clip-index-status", workId],
      enabled: !!workId,
      queryFn: () => apiFetch<ClipIndexStatus>(`/api/clip-index/status?workId=${workId}`),
    });
  }

  export function useClipSearch(workId: string | null, debouncedQuery: string, topK = 20) {
    return useQuery({
      queryKey: ["assets-search", workId, debouncedQuery, topK],
      enabled: !!workId && !!debouncedQuery,
      queryFn: () => apiFetch<ClipSearchResponse>(`/api/works/${workId}/assets/search?q=${encodeURIComponent(debouncedQuery)}&topK=${topK}`),
    });
  }

  export function useBuildClipIndex(workId: string) {
    const qc = useQueryClient();
    return useMutation({
      mutationFn: () => apiFetch(`/api/clip-index/build`, { method: "POST", body: JSON.stringify({ workId }), headers: { "Content-Type": "application/json" } }),
      onSuccess: () => { qc.invalidateQueries({ queryKey: ["clip-index-status", workId] }); },
    });
  }
  ```

## 3. Risks and mitigations

| Risk | Likelihood | Impact | Mitigation |
| --- | --- | --- | --- |
| ViT-B/32 quality is too low for short-video gallery search | Medium | Medium | D1 opt-in to ViT-L/14 via env var; if widespread complaints, flip default in 8.1.5. |
| 5-minute build timeout exceeded for very large libraries (200+ assets, lots of long videos) | Low | Low | Document the limit in README; 8.1.5 adds streaming progress + chunked builds. |
| ffmpeg not installed on a teammate's machine | Low | Medium | `frame_extract.py` returns False on subprocess failure; `build_index.py` skips video assets with stderr warning rather than crashing; image-only indexing still works. |
| faiss-cpu wheel not available for some Python version | Low | Medium | Pinned `faiss-cpu>=1.7.4` is broadly available; stub mode trips on import error and search degrades gracefully. |
| CLIP model weights download fails in restricted network | Medium | High | First-run fetches from HuggingFace; if blocked, the weights can be pre-downloaded into `~/.cache/huggingface/hub/`. README documents the fallback. Stub mode covers the worst case. |
| Search results are wrong because index was built with a different model than search uses | Medium | High | D1 model-mismatch hard error in `meta.json` validation; UI shows "rebuild required" hint when status reports mismatch. |
| Path traversal via malicious workId | Low | Critical | `SAFE_ID` regex enforced in `buildClipIndex`/`searchClipIndex`/`getClipIndexStatus` and at all 3 routes (D9 follow-through). Same regex used elsewhere in `api.ts`. |
